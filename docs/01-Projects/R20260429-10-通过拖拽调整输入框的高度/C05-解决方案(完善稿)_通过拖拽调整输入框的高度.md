## 1. 范围与结论总览

| 编号 | 事项 | 完善后结论 |
|------|------|------------|
| S-01 | 持久化粒度 | **全局一条偏好**（写入 `ClaudianSettings`），与 C02「待澄清」默认建议一致；多标签共用同一拖拽高度基准，避免状态分叉与心智负担。 |
| S-02 | 拖拽改变哪一块几何 | 对 **整块底部组合器**施加 **`min-height`**，锚点元素选 **`.claudian-input-container`**（含队列行、导航行、内层 wrapper、工具栏在内），与用户截图「整张卡片上沿」一致。 |

## 2. 纵向布局与锚点元素（源码依据）

### 2.1 `.claudian-tab-content` 内兄弟顺序

`buildTabDOM`（`src/features/chat/tabs/Tab.ts`）在 **同一 `contentEl`** 下依次为：

1. `.claudian-messages-wrapper` — `flex: 1`、`min-height: 0`（见 `src/style/components/messages.css`），占据中部可伸缩空间。  
2. `.claudian-status-panel-container` — 状态面板占位（可为空）。  
3. `.claudian-input-container` — **底部组合器**，当前 **未**在样式中写死整块「用户偏好高度」，高度来自子元素与 `textarea` 的 JS/`flex` 行为。

拖拽语义：用户拖动 **组合器顶缘**（与消息区/状态区交界），等价于为 **第 3 块**设定不小于基准的 **`min-height`**，从而在竖直方向上占用更多像素；第 1 块随之被挤压（仍为 `flex: 1`）。

### 2.2 `.claudian-input-container` 内部顺序（与截图对照）

同一文件中，组合器内部自上而下为：

- `.claudian-input-queue-row`（队列提示，可为隐藏）  
- `.claudian-input-nav-row`（标签徽章、闪电、`history` 等顶栏图标）  
- `.claudian-input-wrapper`（内层描边卡片）  
  - `.claudian-context-row`（附件芯片等）  
  - `.claudian-input`（`textarea`）  
  - `.claudian-input-toolbar`（在 `initializeInputToolbar` 中挂载，`Tab.ts` 约于工具栏创建处插入）

侧栏主题下，外层组合器另有卡片化样式（`src/style/features/obsidian-layout-claudian-view.css`）。用户感知的「最上方可拖拽边界」对应 **组合器整块的上边缘**；命中区应落在这条边上（§6）。

### 2.3 根容器约束

`.claudian-container` 为列 flex、`height: 100%`、`overflow: hidden`（`src/style/base/container.css`）。输入区增高会减少消息区可视高度，须在方案上限制 **组合器 `max`**，避免消息列表被压到不可用（§5）。

## 3. 现有高度逻辑（不得破坏的行为）

### 3.1 常量（`src/features/chat/tabs/types.ts`）

- `TEXTAREA_MIN_MAX_HEIGHT = 150`  
- `TEXTAREA_MAX_HEIGHT_PERCENT = 0.55`

### 3.2 `autoResizeTextarea`（`Tab.ts`）

逻辑概要：

- 先将 `textarea.style.minHeight` **清空**，再根据内容与 flex 分配高度决定是否设 `min-height`；并设置 `max-height` 为 `max(150px, 55% × .claudian-container 高度)`。

```434:456:src/features/chat/tabs/Tab.ts
function autoResizeTextarea(textarea: HTMLTextAreaElement): void {
  // Clear inline min-height to let flexbox compute natural allocation
  textarea.style.minHeight = '';

  // Calculate max height: 55% of view height, minimum 150px
  const viewHeight = textarea.closest('.claudian-container')?.clientHeight ?? window.innerHeight;
  const maxHeight = Math.max(TEXTAREA_MIN_MAX_HEIGHT, viewHeight * TEXTAREA_MAX_HEIGHT_PERCENT);

  // Get flex-allocated height (what flexbox gives the textarea)
  const flexAllocatedHeight = textarea.offsetHeight;

  // Get content height (what the content actually needs), capped at max
  const contentHeight = Math.min(textarea.scrollHeight, maxHeight);

  // Only set min-height if content exceeds flex allocation
  // This forces the wrapper to grow while letting it shrink when content reduces
  if (contentHeight > flexAllocatedHeight) {
    textarea.style.minHeight = `${contentHeight}px`;
  }

  // Always set max-height to enforce the cap
  textarea.style.maxHeight = `${maxHeight}px`;
}
```

**推论**：用户拖拽设定的「整块组合器」下限必须参与 flex 分配（容器/wrapper `min-height`），否则仅靠空内容时的 `textarea` 高度，无法稳定呈现「拉高空白编辑区」的体验。

### 3.3 CSS 基准（`src/style/components/input.css`）

- `.claudian-input-wrapper { min-height: 140px; }`  
- `.claudian-input { min-height: 60px; flex: 1 1 0; resize: none; }`

这些是 **无用户偏好时** 的排版下限组成部分；**用户偏好不得小于**在无覆盖样式下测得的组合器 **整条盒子高度**（含导航与外层 padding），而不是单独 `140px` 文案（见 §5 `baselineComposerHeightPx`）。

## 4. 推荐实现策略（数据流）

### 4.1 设置字段

在 **`src/core/types/settings.ts`** 的 `ClaudianSettings` 增加可选字段（名称可二选一，实施时择一并全文统一）：

| 候选名 | 语义 |
|--------|------|
| `composerPreferredMinHeightPx` | 用户对 **`.claudian-input-container`** 期望的 **`min-height`（px）**，已钳制；缺省表示「未启用用户拉伸」。 |

在 **`src/app/settings/defaultSettings.ts`** 中 **不显式赋值**（或赋 `undefined`），保证升级用户的默认行为与今日一致。

**加载时规范化**（可在 `main.ts` 的 `loadSettings()` 末尾或专用 `normalizeComposerHeight(settings)`）：剔除 `NaN`、负数；并按 §5 与当前视图/clamp 算法收敛到合法区间（非法则回退 `undefined` 或单次矫正后 `saveSettings()`）。

### 4.2 应用到 DOM

- 对每个 Tab：在布局就绪后（例如视图 `onOpen`、Tab 显示、`requestAnimationFrame` 或 `ResizeObserver`），读取 `plugin.settings.composerPreferredMinHeightPx`。  
- 若存在：设 `dom.inputContainerEl.style.minHeight = `${px}px``（或写入 CSS 变量并由样式引用，便于 Dark/Light 一致）。  
- **下限钳制**：`applied = max(stored, baselineComposerHeightPx)`（§5）。  
- **上限钳制**：`applied = min(applied, composerMaxHeightPx)`（§5）。

拖拽过程中使用临时内联样式同一属性；**mouseup** 后将数值写入 `plugin.settings` 并 **`await plugin.saveSettings()`**（与 `main.ts` 既有持久化路径一致）。

### 4.3 与 `autoResizeTextarea` 的调用顺序

- 施加/修改组合器 `min-height` 之后 **调用一次** `autoResizeTextarea(dom.inputEl)`，避免滚动条与 `max-height` 瞬时错位。  
- 现有 **`input` 监听**（`wireTabInputEvents`）保持不变；无需为拖拽 duplicate 一整套 resize 逻辑。

### 4.4 视图尺寸变化

- 订阅 **`window.resize`** 或对 `.claudian-container` **`ResizeObserver`**：`composerMaxHeightPx` 与 `baselineComposerHeightPx` 可能变化，需重新钳制当前 `min-height`，必要时 **下调到合法上限** 而不静默丢失用户偏好——推荐仍保存「理想」像素值到 settings，仅在应用层 clamp（下次大屏打开自动恢复）。

## 5. `baselineComposerHeightPx` 与上下限公式（可实施定义）

### 5.1 基准最小高度 `baselineComposerHeightPx`

**目的**：满足 C02 E3「默认即最小、不能再缩小」——缩小拖拽终点不得低于 **未施加用户 `min-height` 时**、在同一窗格宽度与当前主题下的组合器高度。

**推荐算法（实现优先级从高到低）**：

1. **测量法（首选）**：在 Tab 首次展示且 **未**设置临时拖拽样式时，对 `dom.inputContainerEl.getBoundingClientRect().height` 取值，并在布局稳定后（`requestAnimationFrame` ×1～2 或 `ResizeObserver` 首轮回调）缓存为 `baselineComposerHeightPx`。队列行展开、Plan mode class、上下文芯片出现时会改变高度——与用户约束的关系：  
   - **保守策略**：基准仅在「典型空闲状态」测量（无队列文案、无附件芯片），写入注释并在验收中说明；缩小拖拽 floor 使用该基准。  
   - **进阶策略（可选）**：当队列行或 context row 显示变化导致基准高于当前用户 `min-height` 时，用 CSS/`min()` 避免布局溢出，或同步抬升有效下限（实施成本高，可作为二期）。

2. **常量兜底**：若测量失败（极少数宿主环境），使用经验常量（例如 **≥220**，需实机校准），并在代码中集中命名 `COMPOSER_MIN_HEIGHT_FALLBACK_PX`。

### 5.2 最大高度 `composerMaxHeightPx`

必须同时满足：

- **textarea 既有上限**：与 `autoResizeTextarea` 一致的整体意图——组合器不应迫使文本区突破 `max(150, 55%·viewHeight)` 所隐含的纵向预算（具体映射到「整块盒子」时可用同一视窗高度比例或略收紧）。  
- **消息区可用性**：建议保证 `.claudian-messages-wrapper` 至少保留 **`messagesMinReservedPx`**（建议默认 **120～160px**，可配置常量），即：

  `composerMaxHeightPx = min(composerMaxFromTextareaRule, tabRoot.clientHeight - statusPanelHeight - messagesMinReservedPx - otherChrome)`

其中 `tabRoot` 宜取 `.claudian-tab-content` 或 `.claudian-container` 的 `clientHeight`，以源码实测为准。

实施时将 **`min(composerMaxHeightPx, baselineComposerHeightPx)`** 在极小窗格下处理得体：若上限低于下限，**以上限为准**并禁用向下拖拽或静默钳制（验收清单 §7）。

## 6. 拖拽交互规格（UX）

### 6.1 命中区

- 在 **`dom.inputContainerEl` 内顶部** 插入专用节点（如 `.claudian-composer-resize-handle`），高度 **4～8px**，全宽，`cursor: ns-resize`，`flex-shrink: 0`。  
- **置于首个子元素位置**，以保证视觉上紧贴组合器顶缘（当前子节点顺序见 §2.2）。  
- **明暗主题**：使用 `var(--background-modifier-border-hover)` 或半透明线与 hover 态区分，避免「看不见把手」。

### 6.2 指针与拖拽

- `mousedown`：记录 **指针 Y** 与 **当前组合器实际高度**（`getBoundingClientRect().height`）。可选：`element.setPointerCapture(pointerId)`（改善指针移出元素后的追踪，宿主支持时）。  
- `mousemove`（绑定 `document`）：  
  - `delta = event.clientY - startClientY`（指针**上移**为负）。  
  - `nextHeight = startHeight - delta`（上移 → 高度增大）。  
  - `nextHeight` 经 §5 钳制后写临时 `min-height`。  
- `mouseup`/`pointerup`/`pointercancel`：落地 settings，移除捕获，`autoResizeTextarea` 一次。

### 6.3 拖拽中的副作用

- 拖拽开始时 **`document.body.style.userSelect = 'none'`**（或等价类），结束后还原，避免误选正文。  
- **不负责**改变 Obsidian 工作区分栏宽度；仅为 Claudian 视图内部纵向分配。

### 6.4 可访问性（最低）

- 手柄 **`role="separator"`**、`aria-orientation="horizontal"`，并 `aria-valuenow` / `aria-valuemin` / `aria-valuemax`（映射钳制后的像素或百分比），便于屏幕阅读器感知存在调节条。

## 9. 用户视角：操作路径推演与衍生问题

以下为 **按时间顺序** 模拟真实用户操作，并标注与实现对标的 **风险与对策**（补充 §5～§7 未展开的边角）。

### 9.1 顺畅路径（期望无摩擦）

| 步骤 | 用户动作 | 预期感知 | 实现注意 |
|------|----------|----------|----------|
| 1 | 首次打开 Claudian 侧栏对话 | 底部输入区高度与今日版本一致 | `composerPreferredMinHeightPx` 缺省，不改 DOM |
| 2 | 将指针移到组合器 **最上沿** | 指针变为纵向调整形；有一条细「分割感」 | 手柄命中区不宜过厚（避免占掉队列行点击区域），也不宜过薄（难点中） |
| 3 | 按住左键 **向上**拖 | 输入区整体变高，消息区随之变矮 | `nextHeight = startHeight - (clientY - startY)`；宜配合 `setPointerCapture` |
| 4 | 松开鼠标 | 高度保持；再次拖动可从当前高度继续 | 写入 settings 并 `saveSettings()`；再调 `autoResizeTextarea` |
| 5 | 关闭 Obsidian 再打开 | 仍为上次高度 | 全局持久化字段读出后施加 `min-height` |
| 6 | 清空输入框或发送消息后输入变短 | 大块留白仍在（用户主动要的「写作台面」） | **勿**在发送成功路径里清除用户偏好；仅清 `textarea` 内容与 resize 逻辑 |

### 9.2 易混淆场景（体验风险）

| 场景 | 用户心理 | 可能的问题 | 对策 |
|------|----------|------------|------|
| **消息列表滚动条** 紧贴组合器上沿 | 想滚消息，结果拖动了高度 | 误触分割线 | 手柄仅限顶缘 **窄条**；消息区 `scroll` 与手柄 **分层**：手柄 `flex-shrink:0` 全宽但高度小；必要时 `pointer-events` 仅在手柄条与队列行之间择一（队列行优先可点击）。 |
| **队列提示行**（`.claudian-input-queue-row`）出现 | 组合器总高度变高 | 用户误以为「偏好失效」或 baseline 漂移 | §8 已列：**floor** 采用空闲态测量；队列展开时有效高度可高于存储值——文档化预期，二期可做动态 floor。 |
| **附件芯片 / 上下文行** 展开 | 同上 | 缩小拖拽终点若仍以空闲 baseline 为 floor，可能出现「当前盒高 > 偏好 min-height」——合法 | 仅需保证 **inline min-height 不大于实际内容所需**；不必强行压扁内容。 |
| **发送长消息后只剩空框** | 「为什么还这么高？」 | 误解 | 一期接受；可选在设置中加「重置输入区高度」或 **双击手柄恢复默认**（见 §9.6）。 |
| **状态面板**（`.claudian-status-panel-container`）展开（如 Bash 输出） | 消息区突然变矮 | 用户误以为拖拽偏好失效 | 纵向顺序为 **消息 → 状态 → 输入**；状态区增高会挤压消息区，与输入 `min-height` **无关**——需在心智上区分「中间动态区」与「底部偏好高度」。 |

### 9.3 与现有「隐藏输入区」流程的交互（必须处理）

`InputController` 在 **权限审批**、**AskUserQuestion** 等流程中会对 **`inputContainerEl`** 执行 **`display: none`** / 恢复（`hideInputContainer` / `restoreInputContainer`，见 `src/features/chat/controllers/InputController.ts`）。

| 风险 | 对策 |
|------|------|
| 用户正在拖拽时弹出内联提问，输入区被隐藏 | **拖拽状态必须结束**：在 `visibilitychange`、`blur`（视图失焦）、以及 **检测到容器 `display:none`** 时走与 `mouseup` 相同的收尾（是否落盘：若已产生有效临时高度则保存，否则丢弃——推荐 **仅在实际变更且指针已按下过时才写 settings**，避免误触）。 |
| `display:none` 导致 `getBoundingClientRect` 为 0 | 布局钳制、基准测量应跳过隐藏帧；恢复显示后 **下一帧**再应用偏好与 `ResizeObserver`。 |

### 9.4 多标签、多窗格与全局偏好

| 场景 | 说明 |
|------|------|
| 同一视图内 **切换 Claudian 标签** | 全局一条 `composerPreferredMinHeightPx`，切换标签后 **应立即应用同一 min-height**（各 Tab DOM 独立，需在 Tab 显示时同步插件设置）。 |
| **两个 Obsidian 窗格** 同时打开 Claudian | 两处共享同一 settings 文件 → **高度偏好一致**；若用户期望「左窄右宽不同高度」，一期不满足——需在文档/FAQ 标明 **「全局像素偏好」**，二期再考虑按叶子维度持久化（储存成本高）。 |

### 9.5 窗口缩放、分辨率与外接屏

| 场景 | 问题 | 对策 |
|------|------|------|
| 用户 **拖窄侧栏** 或 **最大化/还原** | `composerMaxHeightPx` 与 `baselineComposerHeightPx` 变化 | §4.4 `ResizeObserver`：运行时钳制；**磁盘保留用户原始像素**，大屏恢复时自动吃满偏好（已述）。 |
| **笔记本接外接显示器**、或复制 vault 到另一台机器 | 记录的绝对像素可能「过大/过小」 | 一期接受；二期可考虑 **`vh` 比例**或 **`composerPreferredMinHeightRatio`** 并存迁移策略。 |

### 9.6 可选增强（非一期必做）

| 项 | 说明 |
|----|------|
| **双击手柄恢复默认** | 清除 `composerPreferredMinHeightPx`（或设为 `undefined`）并去掉 inline 样式，回归 CSS 纯默认。 |
| **键盘调节** | 在 `role="separator"` 基础上实现 ArrowUp/ArrowDown（可选），步进约 10px；需防与消息区 vim 导航冲突（仅在手柄获得焦点时吞键）。 |
| **触控** | 窄条对手指不友好；二期加大触控命中或长按进入拖拽模式。 |

### 9.7 性能与事件 hygiene

| 项 | 说明 |
|----|------|
| `mousemove` 高频 | 使用 **`requestAnimationFrame` 节流**写样式，避免布局抖动。 |
| 拖拽结束 | **务必**移除 `document` 级监听，避免泄漏与「幽灵拖拽」。 |
| `pointercancel` | 触控板手势、系统抢占时需与 `mouseup` 同等处理。 |
