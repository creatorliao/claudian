# C02 问题报告：通过拖拽调整输入框高度

## 1. 背景与现象

在部分使用场景下，用户需要在单次会话中编写**较长的指令或上下文**，当前侧栏对话底部的输入区域高度主要由布局与「随内容增高」逻辑决定，**无法在交互上一次性拉高整块输入区**，只能靠输入换行被动撑高（且受最大高度比例限制）。用户期望像调整 Obsidian 分割窗格一样，通过**拖动输入区上边缘**主动设定舒适的输入高度。

## 2. 期望行为

| 编号 | 期望 |
|------|------|
| E1 | 鼠标置于输入区域**上方边界**（或与窗格分隔类似的拖拽条）时，可进入**纵向拖拽**，增大或减小输入区占用高度。 |
| E2 | **松开鼠标后**记住本次设定的高度（或等价的状态），下次打开插件视图时恢复。 |
| E3 | **默认高度为最小值**；不得继续缩小至低于该最小值，避免因窄窗格等情况下过度压缩输入区导致可用性问题。 |
| E4 | 交互感受接近 Obsidian **拖拽调整窗格大小**（明确命中区、拖拽中光标与反馈）。 |

## 3. 复现 / 现状说明

非 Bug：当前实现即为「固定最小布局 + `textarea` 随内容增高」，未提供用户可控的基准高度或拖拽柄。任意会话中均可观察到无法仅靠拖拽上边框预留大块空白输入区。

## 4. 影响范围与优先级

- **范围**：主对话视图（`.claudian-input-container` / `.claudian-input-wrapper` 一带）；与消息列表共享纵向空间。
- **严重程度**：体验增强（非阻断性）；对长指令、多段粘贴用户影响较大。
- **优先级**：按产品排期；实现时需与现有 `autoResizeTextarea` 行为协调（见方案文档）。

## 5. 环境

- **插件**：Claudian（Obsidian 插件）。
- **典型环境**：桌面端 Obsidian（Windows / macOS）；移动端若有差异化需在方案阶段注明。

## 6. 待澄清（可选）

- [ ] 拖拽调整的高度是否按「标签页」区分，还是全局唯一偏好（默认建议：**全局**，与设置持久化一致）。
- [ ] 「最小高度」是否严格等于当前 CSS 默认值（约 140px 外层 wrapper 注释所述），还是需在设置中可配置。
