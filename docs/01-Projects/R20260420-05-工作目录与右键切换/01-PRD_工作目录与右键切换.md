# PRD：Claudian 工作目录（Workspace）与右键切换

> **版本**：以本仓库当前实现为准（文档编写：2026-04-20）  
> **范围**：Vault 内子文件夹作为 Claude Agent SDK 的 **effectiveCwd（工作目录）**；Obsidian 文件树右键设置；重置为 Vault 根；多 Tab 各自工作空间快照。

---

## 1. 背景与目标

### 1.1 问题

Vault 内常有多个子项目（如 `projects/A`、`projects/B`）。若 Agent 始终以 Vault 根为 `cwd`，相对路径、Bash `pwd`、工具解析默认目录会与用户心智不一致。

### 1.2 目标

1. 用户可将 **Vault 内某一文件夹** 设为 Claudian 的**默认工作空间**（持久化），**新建聊天 Tab** 时继承该目录为有效 `cwd`。
2. 用户可通过 **文件树文件夹右键** 快速写入上述持久化配置，并得到操作反馈。
3. 用户能**一眼看到**当前上下文对应的目录（聊天标题区副标题 + Ribbon 提示）。
4. 用户可**一键重置**为 Vault 根（清空持久化，并将当前视图内所有 Tab 的工作空间快照清空）。

### 1.3 非目标（当前实现不包含）

- **不在**插件设置页提供「工作空间」文本框（历史 C01 曾含 R04；后因配置页体验问题移除，见 R20260223-01 **C05** 说明）。工作空间仅通过 **右键** 与 **重置** 修改。
- 不支持将 Vault **外部**路径设为工作空间（必须在 Vault 内且为**目录**）。

---

## 2. 术语

| 术语 | 含义 |
|------|------|
| **持久化工作空间** | 存在插件 `data.json` 的 `workspace` 字段，值为 **Vault 相对路径**（如 `notes/proj`），空字符串表示 Vault 根。 |
| **Tab 工作空间快照** | 每个 `TabData.workspace`：创建 Tab 时根据持久化解析得到的 **绝对路径**，或 `null`（表示使用 Vault 根）。**运行时**仅「重置工作空间」会批量置 `null`。 |
| **有效工作目录 effectiveCwd** | `resolveWorkspacePath(tab.workspace, vaultPath) ?? vaultPath`，作为 `queryOptions.effectiveCwd` 等传入 Agent SDK。 |

---

## 3. 用户故事与行为规格（与代码一致）

### 3.1 文件夹右键「设为工作空间」

- **前置**：用户在文件树对 **文件夹**（`TFolder`）右键。
- **操作**：选择「设为工作空间」（文案来自 i18n `contextMenu.setWorkspace`）。
- **系统行为**：
  1. 校验：`path.join(vaultPath, file.path)` 必须在 Vault 内（`isPathWithinVault`）。
  2. 持久化：`storage.setWorkspace(file.path)`，即将 **Obsidian 相对路径**（与 `TFile.path` 一致）写入 `data.json`。
  3. 提示：`Notice` 成功；更新 Ribbon 的 `setTooltip`（`updateRibbonTooltipFromStorage`）。
- **刻意不做的行为（定稿）**：**不修改**任何已打开 Tab 的 `tab.workspace`。因此**当前 Tab 的 Agent `cwd`、标题副标题**在右键后**可能仍对应旧快照**，直到用户新建 Tab、切换逻辑触发刷新、或重启后恢复状态等场景。**新建 Tab** 会读取最新持久化并解析为新的 `tab.workspace`。
- **与历史 C01 的差异**：C01 曾写「点击后当前 Tab 立即切换」；**当前产品与 C05/代码**为「只改持久化，已开 Tab 不变」。

### 3.2 重置工作空间

- **入口**：聊天标题区 **folder-x** 按钮；命令面板 **Reset workspace**（`reset-workspace`）。
- **行为**：`plugin.resetWorkspaceAndNotify()`：`storage.setWorkspace('')`；对所有已打开的 `ClaudianView` 调用 `resetWorkspaceToVaultRoot()`，将每个 Tab 的 `workspace` 置 `null`；`Notice`；刷新 Ribbon。

### 3.3 新建 Tab 的继承规则

- 创建 Tab 时：`initialWorkspace = resolveWorkspacePath(storage.getWorkspace(), vaultPath)`（若持久化为空则得到 `null`）。
- 解析失败（路径不存在、非目录、越出 Vault）时 `resolveWorkspacePath` 返回 `null`，效果等同 Vault 根。

### 3.4 会话恢复（Tab 持久化）

- `tabManagerState` 中每个 Tab 可存 **`workspace` 绝对路径快照**；恢复时传入 `createTabWithWorkspace(..., persistedWorkspace)`，优先用快照而非当前全局持久化，保证「上次关闭前每个 Tab 的 cwd」可还原。

### 3.5 可见性

- **聊天标题旁副标题**：展示当前**激活 Tab** 的有效工作区（`formatWorkspaceDisplayShort` / `workspace.vaultRoot`）；完整绝对路径放在 `title` tooltip。
- **Ribbon**：未打开聊天时也应能通过 tooltip 看到**持久化**解析后的工作区简称（与副标题数据源可能不一致——见实现方案中的「Ribbon vs 副标题」）。

### 3.6 关联能力（工作空间收窄）

- **打开页签 → 内部上下文胶囊**：仅收录 **落在当前 Tab 工作空间内**的打开文件（`FileContextManager` + `isInWorkspace`），避免引用 Agent 无法读的路径。
- **Bang Bash（! 模式）**：`getCwd` 每次执行前动态解析当前 Tab 的 `resolveWorkspacePath(tab.workspace)`。

### 3.7 多页签与工作空间：如何关联、会不会搞混

#### 3.7.1 两层数据，职责不同

| 层级 | 存什么 | 和「第几个页签」的关系 |
|------|--------|------------------------|
| **全局持久化**（`data.json` → `workspace`） | 一份 **Vault 相对路径**，表示「**接下来新建**的聊天页签默认用哪个子目录」 | **不**按页签编号存储；所有**新开的**页签在创建那一刻都会读它并解析成绝对路径写入该页签自己的快照 |
| **每页签快照**（`TabData.workspace`） | 该页签专属的 **绝对路径或 `null`**（Vault 根语义） | **每个页签一条**；发消息、`effectiveCwd`、文档胶囊过滤、Bang Bash 的 `cwd` 都只看**当前激活页签**的这一条 |

因此：**不是**「全局只有一个 cwd 全插件共用」；也**不是**「右键一改所有页签立刻一起变」。真实模型是 **「每页签自带 cwd 快照 + 全局只负责默认值」**。

#### 3.7.2 页签生命周期里快照从哪来

1. **用户点击新建页签**：`TabManager` 调用 `storage.getWorkspace()`，经 `resolveWorkspacePath` 得到 `initialWorkspace`，写入**新** `TabData.workspace`（解析失败则为 `null` → Vault 根）。
2. **重启/恢复布局**：`tabManagerState` 里每个页签可带当时保存的 **`workspace` 绝对路径**；恢复时**优先用这条快照**，而不是用当前全局 `data.json.workspace`——所以**不同页签可以长期对应不同子目录**（例如 Tab1 在 `proj/A`、Tab2 在 `proj/B`）。
3. **右键「设为工作空间」**：只改 **全局** `data.json`；**已打开页签的快照不变**。因此会出现：全局默认已是 B，但旧页签仍在 A 下干活——这是**当前定稿行为**，不是偶然 Bug。
4. **重置工作空间**：清空全局 + 把**当前 Claudian 视图里所有页签**的 `workspace` 一律置 `null`（多视图/多窗口时，插件会对**每个**已打开的 Claudian 视图执行同一套重置逻辑）。

#### 3.7.3 会不会「搞混」？分两层说

- **执行层（发消息、工具、SDK）**：**一般不会混**。每条链路都绑定在**当前 Tab 对象**上：`InputController` 的 `getEffectiveCwd`、`ClaudianService.ensureReady` 用的都是**该 Tab** 的 `tab.workspace`。你切换到 Tab2 再发送，用的就是 Tab2 的快照，不会偷偷用 Tab1 的目录。
- **认知层（用户以为自己在哪）**：**有可能晕**。典型情况：
  - **Ribbon 提示**紧跟 **全局持久化**；**标题旁副标题**跟 **当前激活页签的快照**。右键换了全局默认后，Ribbon 已显示新文件夹，但当前页签副标题仍是旧的，直到切换页签/新建页签等触发刷新——两处信息可能**暂时不一致**。
  - 多个页签若在不同时间创建或从持久化恢复，**本来就允许**各自 `cwd` 不同；若用户以为「全局只有一个工作空间」，会误以为程序乱了。

#### 3.7.4 使用建议（写进 PRD 侧的用户预期）

- 要**立刻**在某目录下对话：依赖「**新建页签**」（或确认当前页签副标题/tooltip 已是该目录）。
- 要**并行**两个项目：用**两个页签**，各自在合适时机新建，或依赖恢复布局保留各自快照；不要只靠 Ribbon 判断当前发送落在哪。
- **重置**会一次性清空**本视图所有页签**的目录快照，并行任务时注意是否都要回 Vault 根。

## 6. 不想折腾「全局 + 页签快照」时：怎么简单又对准？

复杂感主要来自 **两层状态**（持久化默认值 vs 每页签快照）以及 **Ribbon 与副标题数据源不同**。若希望**少记规则、又尽量让「我在哪个目录下说话」一眼对准**，可按优先级选用下面几条（**不改代码**与**改 fork** 分开写）。

### 6.1 零改代码：三条习惯就够准

| 做法 | 为什么能对准 |
|------|----------------|
| **只认「标题旁副标题 + 鼠标悬停看完整路径」** | 它和 **当前页签的 `effectiveCwd`** 一致；**不要**单靠 Ribbon 判断「这一页正在用哪」。 |
| **右键换文件夹之后，若要立刻在新目录下聊：请「新建页签」** | 新页签才会读最新持久化；**「新对话」同页签不会改 `tab.workspace`**， cwd 仍是旧的。 |
| **并行项目时：一页签一项目，或把「最大页签数」调到 1** | 页签数少，全局默认与当前页签更容易一致；多页签时自己约定「哪个 Tab 管哪个文件夹」。 |

一句话：**以副标题为准；换目录要立刻生效 → 新开页签；少开页签就少混。**

### 6.2 仍想多页签并行，又不想记两层：依赖「恢复布局」时要知情

重启后每个页签会从 **`tabManagerState` 里各自存的绝对路径** 恢复，**不一定**等于当前全局 `data.json.workspace`。若你希望「永远跟全局右键一致」，需要在习惯上 **改完右键后新建页签** 或 **接受多页签各自记忆**（见 §3.7）。

### 6.3 愿意改 fork：从根上变「简单模型」（概要）

若你在自己的分支上愿意改产品行为，可把模型收成下面两类之一（实现要点见 [02-实现方案 §7](./02-实现方案_代码事实与复现步骤.md)）：

1. **「全局即真理」**：右键设置后，**同步**更新所有已打开页签的 `tab.workspace`（并视需要 `ClaudianService` 重连/更新 `effectiveCwd`）。用户只认一个目录，代价是会话与进行中的工具状态可能被打断，需评估。
2. **「单工作空间 + 单页签」**：限制 `maxTabs = 1`（或隐藏多页签），并可选在右键后只更新当前激活页签；心智上接近「一个聊天 = 一个 cwd」。

当前上游/本仓库**默认**是 **「默认只影响新建页签 + 每页签可不同」**，偏并行与安全；要简单对准，**优先用 §6.1 习惯**；要产品级简单，再走 §6.3 改造。
