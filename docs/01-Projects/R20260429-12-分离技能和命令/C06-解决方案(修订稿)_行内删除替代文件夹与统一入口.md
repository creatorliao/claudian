## 1. 目标摘要

| # | 目标 |
|---|------|
| 1 | **配置页预览列表**与 **「查看全部」列表**：每条卡片 **右侧主操作** 由「文件夹」改为 **「删除」**（图标 `trash-2`），点击后 **确认**，然后删除对应磁盘实体并重扫缓存。 |
| 2 | **命令**：删除单个命令文件。**技能**：删除技能目录（现有存储语义对齐 SkillStorage）。 |
| 3 | **不区分来源标签**：同一交互不因「本库」或「本机共用」而改变「是否为删除」；必要时在同一确认框内区分风险提示文案。 |
| 4 | **仅此一处删除**：移除或收敛其它位置上冗余的「删除命令/技能」按钮（含此前并排的第二枚垃圾桶、仅灰色的占位垃圾桶等），避免多套入口。 |
| 5 | **「打开文件夹」能力**：若产品仍需访问磁盘路径，改为 **次级入口**（例如条目右键菜单、分区顶部按钮栏、`[`了解更多`]卡片文案外链`**不推荐作为主交互**，或由顶部已有「打开本库 / 本机×目录」按钮保留）；本文默认 **列表行不再承担 reveal**。 |

## 2. 交互细则

### 2.1 确认对话框（友好）

- 标题简短：**删除自定义命令** / **删除技能**（或与现有 i18n key 对齐）。
- 正文：
  - 指明 **`/名称`**；
  - **命令**：说明删除文件；
  - **技能**：说明删除**文件夹及其内容**；
  - **用户主目录条目**：追加一句固定风险提示（示例）：「该项位于本机共用目录，删除将影响电脑上依赖同一文件的 Claude Code 环境。」（最终实现用语须经母语校对）。
- 按钮：**取消**（默认焦点可按产品设计微调）、**删除**（destructive）。
- 使用既有 **`confirmDelete`** 或封装等价 **`ConfirmModal`**，不与原生 `confirm()` 混用。

### 2.2 删除成功后

与 **[C05 §3](C05-解决方案(完善稿)_分离技能和命令.md)** 对齐：`catalog.refresh()` → `listVaultEntries()` → `render()` → `notifyVaultSlashCommandsMutated()`，必要时 Notice。

## 3. 技术方案

### 3.0（必选）实施时必须移除的旧逻辑 —— 与最新方案冲突清单

下列为当前代码路径中已存在、但与 **§1「单一删除入口」「文件夹改删除」「本库与本机共用同一删除语义」** 不一致的实现。**正式编码时应一并拆除**，避免新旧两套删除语义并存（否则会出现「一行两个垃圾桶」「文件夹仍是 reveal」「本机只能灰显」等与文档不符的行为）。

| # | 旧逻辑（错误或过时的语义） | 典型位置 | 处理方式 |
|---|---------------------------|----------|----------|
| A | 行尾 **文件夹图标**（`folder-open`）调用 **`revealSlashEntryPath`**，语义为「在文件管理器中打开」 | `SlashCommandSettings.ts` → `renderCommandRow` 内 `if (cmd.slashFileProvenance)` 块 | **移除**该行按钮及监听；行尾主操作只保留 **删除**（见 §3.1）。若需浏览磁盘路径，依赖弹窗顶部「打开本库/本机×目录」按钮（§1 目标 5）。 |
| B | 与文件夹 **并排的第二枚垃圾桶**：仅当 **`cmd.isDeletable`** 为真时可点删除 | `renderCommandRow` 内 `if (cmd.isDeletable) { … confirmDelete … deleteCommand }` | **移除该套「第二按钮」分支**，改为 **一枚删除按钮** 走 §3.2 统一 API；不再用 `isDeletable` 在 UI 上分叉两套图标。 |
| C | **`user-home` 专用分支**：灰色禁用垃圾桶 + `deleteUnavailableUserHomeAria` / `Hint` | `renderCommandRow` 内 `else if (cmd.slashFileProvenance === 'user-home')` | **整段移除**。该分支是为「禁止插件删本机」打的补丁，与 **§1 目标 3**（本机条目也可删，确认框提示风险）冲突。 |
| D | **Catalog 曾禁止删用户主目录**：`slashCommandToEntry` 中 **`isDeletable: cmd.source !== 'sdk' && !fromUserHome`**；**`deleteVaultEntry`** 对 `user-home` **抛错** | `ClaudeCommandCatalog.ts` | **按 §3.2 重写**：文件型条目删除统一走扩展后的删除路径；否则 UI 无法实现「不区分来源」。 |
| E | **样式**：`.claudian-settings-action-btn:disabled` 专为灰色垃圾桶加的弱化样式 | `src/style/settings/base.css`（若有） | 若移除禁用分叉后 **无其它禁用图标按钮**，可删冗余规则；若有通用用途则保留。 |

**自检**：实施后全局检索 `folder-open`、`revealInFileManagerAria`（行内）、`deleteUnavailableUserHome`、`else if.*user-home.*delete`、`isDeletable`（若仅用于 UI 分叉可考虑收窄或删除），确保 **唯一删除路径** 与文档一致。

---

### 3.1 移除「在行上用文件夹扮演 reveal」

- **`renderCommandRow`**（预览列表与「查看全部」共用）：去掉 **`folder-open`** 作为主按钮的逻辑。
- **单一按钮**：删除（唯一与服务语义对齐的行尾快捷入口）。

### 3.2 删除逻辑统一到 Catalog / Storage

- **当前**：`deleteVaultEntry` 对用户主目录条目 **抛错**，配合 `isDeletable: false`。
- **修订**：
  1. 提供 **单一删除入口方法**（命名可为 `deleteSlashAsset(entry)` 或扩展 `deleteVaultEntry` 语义），内部根据 `slashFileProvenance` 与 `kind` 路由到：
     - **Vault**：沿用现有 `SlashCommandStorage.delete` / `SkillStorage.delete`；
     - **User-home**：在同存储层实现「解析绝对路径 → 删除文件或目录」，须 **跨平台路径**（`path`/`fileURLToPath`），并遵守仓库 Windows/macOS 约束。
  2. **`slashCommandToEntry`** 或等价处：将 **`isDeletable`** 对文件型条目统一为 **true**（或与「可解析出可控路径」等价），不再因 user-home 禁用删除。
  3. **SDK / 非文件条目**：仍保持不可删（若有混入）；设置列表当前以 `listVaultEntries()` 为准。

### 3.3 UI 收敛「其它位置不应有删除」

实施后自检清单（检索关键字示例：`deleteVaultEntry`、`deleteCommand`、`trash-2`、`claudian-settings-delete-btn`）：

| 区域 | 动作 |
|------|------|
| `SlashCommandSettings` / `SlashCommandsFullListModal` | **仅**保留 **一枚**行内删除按钮（并移除 §3.0 表 A/B/C 所述旧分支） |
| 新增 / 编辑 `SlashCommandModal` | **不包含**删除入口（编辑已有若曾计划删除，仍以列表为准） |
| 对话侧斜杠下拉等 | **不包含**删除命令/技能配置 |

若存在历史残留的「第二枚垃圾桶」或「灰色占位垃圾桶」，**删除相关分支**（见 **§3.0**），避免重复。

---

### 3.4 实现顺序建议（与 §3.0 对齐）

1. **先数据层**：完成 §3.2（Catalog/Storage 对用户主目录可删），避免先做 UI 仍调用旧 `deleteVaultEntry` 抛错。
2. **再 UI**：按 §3.0 表删除 `folder-open`、`isDeletable` 分叉删除、`user-home` 灰按钮分支，收口为一枚删除。
3. **最后**：清理无用 i18n（如仅服务于灰按钮的键）、样式与 CHANGELOG 条目中与「灰垃圾桶」不一致的描述。

## 4. 风险与合规

| 风险 | 缓解 |
|------|------|
| 删除用户主目录资产影响其它应用 | 确认框明确文案；可选二次勾选「我已知晓」（若用户反馈仍误触再加） |
| 路径解析失败 | 删除失败 Notice；不落库状态不一致 |
| 目录非空 / 权限 | 捕获异常并中文友好 Notice |

## 5. i18n 与无障碍

- **删除按钮** `aria-label`：统一为「删除」或「删除此项对应的磁盘文件或文件夹」，不再使用「在文件管理器中显示」。
- 移除或替换依赖「仅 reveal」的旧提示字符串。
- 保留顶部工具栏「打开本库 / 本机 × 目录」按钮文案（仅来自 **[C06](C06-解决方案(修订稿)_行内删除替代文件夹与统一入口.md)** 若保留次级入口）。
