## 1. 已定产品结论（回应待定项）

### 1.1 「查看全部」计数与入口 — **分开**

- **命令**：仅统计自定义命令条数；按钮文案「查看全部（n₁）」只出现在命令预览区，弹窗标题示例：**「全部自定义命令（n₁）」**。
- **技能**：同上，独立 **n₂** 与独立弹窗。
- **理由**：用户在心理上先做类型选择（写脚本快捷 vs 技能包），合并计数会令「我为什么看到 56 条」难以拆解；分开后责任边界清晰。

### 1.2 「系统」与「配置」— **不用字面「系统」「配置」作顶层标题**

- **不推荐**：在设置页增加抽象的「系统」「配置」二级菜单——多数用户不会联想到「本库磁盘文件 vs 运行时」。
- **推荐**：沿用并已存在的 **`本库` / `本机共用`** 角标说明数据来源；在首次触及「本机共用」行的场景辅以一种简短说明（tooltip 或描述一句）：**在本插件内只能浏览与定位文件夹，修改请到文件管理器**。
- **运行时/SDK**：对话 `/ ` 下拉可能包含探测命令；**设置页仍以 `listVaultEntries()` 文件清单为准**。不设「系统」专区；若日后要把运行时条目放进设置页，再单列「运行时（只读）」分组并单独 PR。

### 1.3 SDK / 运行时条目 — **本迭代仍不进设置列表**

- 避免与文件 CRUD 混在同一列表造成「能删吗」误解。
- 若用户问到「为什么列表里没有某项但能 `/ `」——文档 FAQ：**下拉可由运行时合并；此处只管磁盘上的命令与技能。**

## 2. 产品经理审视：漏洞与加固

| 风险 | 说明 | 处理 |
|------|------|------|
| **垂直高度增加** | 拆成两节后预览总行数上限仍为 3+3，反而可能比原先 5 条混合略高一行标题成本 | 已接受；用小标题分隔换取可读性 |
| **两处刷新** | 命令区与技能区各有刷新按钮，易误以为「只刷新一类」 | **语义**：刷新 = **重新扫描磁盘清单（两类一致更新）**；两处刷新调用同一 `loadAndRender` / catalog，两边实例各自 `render`，结果一致 |
| **两处「+」** | 可能加成命令时点错区域 | **默认类型**：命令区的「+」打开新建命令；技能区的「+」打开新建技能（下拉默认值对齐），仍可切换类型以防进阶用法 |
| **删除对象表述** | 技能对应文件夹，命令对应文件 | 确认框正文区分「命令文件」与「技能文件夹」 |
| **本库条目才可删** | 仅 `isDeletable` | 保持现有逻辑；无确认框对本机条目误删路径 |
| **删除后不刷新运行时缓存** | 可能与下拉短暂不一致 | 删除成功后 **`await catalog.refresh()`** 再 `reloadCommands`，并 **`notifyVaultSlashCommandsMutated`** |
| **空列表** | 一侧为空另一侧有时用户不知所措 | 分区内空态：命令侧 / 技能侧各有一句指向「+」的引导 |

## 3. 技术摘要（实施对齐）

| 项 | 说明 |
|----|------|
| 预览上限 | 命令 **3** 条、技能 **3** 条 |
| `SlashCommandSettings` | **单一实例**内渲染「自定义命令」「技能」两个分区（`renderKindSection`），共享一次 `listVaultEntries()`，避免分区各自刷新导致状态不一致 |
| 「查看全部」 | `SlashCommandsFullListModal` 仅传入该类条目；标题与总计按类型切换 i18n key |
| 分组标题 | 分区标题由 `SlashCommandSettings` 内 `Setting.setHeading()` 输出；`ClaudeSettingsTab` 仍保留总标题「命令与技能」与来源一行 |
| 刷新 | 任一分区点击刷新 → `catalog.refresh()` → `loadAndRender()` → **两侧列表一并重绘** |
| 删除 | `confirmDelete(app, message)`（复用 `src/shared/modals/ConfirmModal.ts`）；通过后执行删除 → `refresh()` → 重绘 |
| 角标 | 命令分区**不**显示「技能」徽章；技能分区亦不重复徽章 |
