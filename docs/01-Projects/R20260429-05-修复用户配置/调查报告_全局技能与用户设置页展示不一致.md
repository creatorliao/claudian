# 调查报告：全局 `~/.claude/skills` 与用户配置页「命令与技能」展示不一致

## 1. 背景与现象

用户在**用户主目录**下创建了多处 **Agent Skills**（例如 `~/.claude/skills/...`），并启用了 Claudian 中的 **「加载用户 Claude 设置」**（UI 文案说明会加载 `~/.claude/settings.json`）。预期在 **Claude 提供商设置页的「命令与技能」列表** 中看到与全局技能、用户侧命令一致的内容，但实际列表**不包含** `~/.claude/skills` 下的技能，且与用户直觉中的「加载用户配置 = 把用户目录下的技能也列出来」不符。

本报告基于当前仓库实现做**代码级根因**说明，不作产品承诺；后续是否改为合并展示、只读展示或文档澄清，由需求/方案阶段决定。

## 2. 用户预期与实现差异（结论摘要）

| 维度 | 用户常见预期 | 当前实现 |
|------|--------------|----------|
| 「命令与技能」列表的数据来源 | 含**库内**与**用户主目录**（`~/.claude`）下的技能/命令 | **仅库（Vault）**内相对路径 `.claude/skills`、`/.claude/commands` 下的文件 |
| 「加载用户 Claude 设置」开关 | 打开后配置页应反映用户目录下的技能等 | 仅影响向 **Claude Agent SDK** 传入的 `settingSources`（是否合并 **用户级** `settings.json`），**不**改变配置页列表的数据源 |

**根因可归纳为两点：**

1. **设置页列表**走的是 `listVaultEntries()`（库文件扫描），**从不**读取 `~/.claude/skills`。
2. **`loadUserClaudeSettings`** 与 **用户级 `settings.json`** 对齐，**不**等同于「把用户目录技能列进设置 UI」；聊天侧运行时可能通过 SDK 发现更多命令，但设置 UI 未与之一致。

## 3. 代码路径说明

### 3.1 设置页「命令与技能」只加载库内条目

`SlashCommandSettings` 初始化时只调用 `catalog.listVaultEntries()`：

```337:345:src/providers/claude/ui/SlashCommandSettings.ts

    this.render();
  }
```

`listVaultEntries` 合并的是库内的命令存储与技能存储，均来自**同一 Vault 适配器**上的相对路径：

```113:117:src/providers/claude/commands/ClaudeCommandCatalog.ts
  async listVaultEntries(): Promise<ProviderCommandEntry[]> {
    const commands = await this.commandStorage.loadAll();
    const skills = await this.skillStorage.loadAll();
    return [...commands, ...skills].map(slashCommandToEntry);
  }
```

应用内该开关映射为 `loadUserClaudeSettings`（并归一化为 `getClaudeProviderSettings(...).loadUserSettings`），用于 **SDK 查询选项**中的 `settingSources`：在打开为 `true` 时使用 `['user', 'project']`，否则仅 `'project'`。例如冷启动探测命令时：

### 3.4 补充：`listDropdownEntries` 与设置页列表不是同一路径

下拉/运行时侧 `listDropdownEntries` 会优先使用 SDK 侧的 `sdkCommands`（若已通过会话或 probe 填充），仅在**空**时回退到 `listVaultEntries()`：

```78:93:src/providers/claude/commands/ClaudeCommandCatalog.ts
  async listDropdownEntries(context: { includeBuiltIns: boolean }): Promise<ProviderCommandEntry[]> {
    void context;
    // SDK commands already include vault commands/skills (the SDK scans
    // .claude/commands/ and .claude/skills/ internally). No file scan needed.
    // When the cache is empty (cold start, no active runtime), probe the SDK.
    if (this.sdkCommands.length === 0 && this.probe) {
      await this.ensureProbed();
    }
    const runtimeEntries = this.sdkCommands
      .filter(cmd => !BUILTIN_HIDDEN_COMMANDS.has(cmd.name.toLowerCase()))
      .map(slashCommandToEntry);
    if (runtimeEntries.length > 0) {
      return runtimeEntries;
    }
    return this.listVaultEntries();
  }
```

**设置页**始终调用 `listVaultEntries()`（见 3.1），**不会**用 `listDropdownEntries()`。因此即使用户在聊天里能触发更多由 SDK 解析出的命令，**配置页列表仍可能只显示库内文件**，与全局技能是否被 CLI 加载**解耦**。

## 4. 与 Codex 侧的对比（便于移植思路）

Codex 技能存储存在 **vault + home** 双适配器用法（`CodexSkillStorage`），而 **Claude 侧技能/命令存储当前仅为 vault 单源**。若产品目标是「设置页可见用户级技能」，需在架构上为 Claude 侧增加对 `~/.claude/skills`（及如需则 `~/.claude/commands`）的只读或可编辑策略，并明确与 SDK `settingSources` 的边界；本报告仅记录现状差异。

## 5. 调查结论（根本原因）

1. **配置页列表数据源设计为「仅 Obsidian 库内的 `.claude` 目录」**，`~/.claude/skills` 不在扫描范围内。
2. **「加载用户 Claude 设置」** 只控制 **SDK 是否合并用户级 `settings.json`**，**不包含**「在设置 UI 中列举用户主目录技能」的实现。
3. **设置页与下拉列表使用了目录不同的 API**（`listVaultEntries` vs `listDropdownEntries`），易造成「运行时里能用到、页面上看不见」的体感差异。

## 6. 建议的后续工作（非方案定稿）

- **文档/文案**：在「命令与技能」区块增加简短说明：列表管理的是**当前库**下 `.claude/commands` 与 `.claude/skills`；用户主目录技能由 Claude Code/SDK 按官方规则加载，**不一定**出现在此列表。
- **产品/开发**：若需统一体验，可立项评估：设置页是否只读展示用户级技能、是否禁止编辑删除、是否与 `listDropdownEntries` 合并去重等（需单独方案与测试）。
