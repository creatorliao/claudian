# 调查报告：Claude「命令与技能」何时刷新、逻辑是什么，以及为何没有「刷新」按钮

## 1. 调查范围

- **Claude 提供商**下：设置页中的 **命令与技能**（`SlashCommandSettings`）、对话输入栏 **斜杠下拉**（`ClaudeCommandCatalog` + `TabManager`）。
- 对比：**子代理**（`AgentSettings`）、**插件**（`PluginSettingsManager`）、**Codex 技能列表**（`CodexSkillSettings`）的刷新按钮与刷新语义。

**结论摘要**：设置页列表在 **构造时** 与 **本 UI 内增删改** 后会重读目录；**无**文件监听，**无**头部刷新按钮；公开的 `SlashCommandSettings.refresh()` **未被任何地方调用**。对话下拉在每次取条目时会尽量 **拉取 SDK 支持的命令** 并写入 catalog，与设置页列表 **不是同一条刷新链路**。

## 2. 设置页「命令与技能」列表：何时更新

### 2.1 数据从哪来

`SlashCommandSettings` 在 `loadAndRender` 中调用 `catalog.listVaultEntries()`（实现上已合并库内与可选的用户主目录，见 `ClaudeCommandCatalog`）。

```370:377:src/providers/claude/ui/SlashCommandSettings.ts
  private async loadAndRender(): Promise<void> {
    if (!this.catalog) {
      this.renderUnavailable();
      return;
    }

    this.commands = await this.catalog.listVaultEntries();
    this.render();
  }
```

### 2.2 触发 `loadAndRender` 的时机

| 时机 | 说明 |
|------|------|
| **组件构造** | `constructor` 末尾 `void this.loadAndRender()`，每次设置页渲染出新的 `SlashCommandSettings` 实例时都会重新扫盘。 |
| **本页内增删改/转换** | `reloadCommands()` 再次 `listVaultEntries()` 后 `render()`。 |
| **`refresh()` 方法** | 仅 `void this.loadAndRender()`，等价于再扫一盘。 |

```623:625:src/providers/claude/ui/SlashCommandSettings.ts
  public refresh(): void {
    void this.loadAndRender();
  }
```

### 2.3 工程中谁调用了 `refresh()`

当前仓库内 **无任何引用** 调用 `SlashCommandSettings.refresh`（`ClaudeSettingsTab` 只 `new SlashCommandSettings(...)`，不保存实例引用）。因此用户 **不能** 通过按钮触发与 `refresh()` 等价的「仅重读列表」。

### 2.4 间接「刷新」：整页 `redisplay`

例如修改「命令与技能的来源」下拉后会 `context.redisplay()`，整页 Claude 提供商会重画，**新的** `SlashCommandSettings` 会再次构造并 `loadAndRender`，列表会反映新的 `slashAssetScope`。

```185:199:src/providers/claude/ui/ClaudeSettingsTab.ts
        .onChange(async (value) => {
          updateClaudeProviderSettings(settingsBag, {
            slashAssetScope: value as 'vault-only' | 'vault-and-user-home',
          });
          await context.plugin.saveSettings();
          context.redisplay();
        });
    });

  const slashCommandsContainer = container.createDiv({ cls: 'claudian-slash-commands-container' });
  new SlashCommandSettings(
    slashCommandsContainer,
    context.plugin.app,
    claudeWorkspace.commandCatalog,
  );
```

### 2.5 不会自动刷新的典型场景

- 设置页 **保持打开**，用户 **在外部编辑器** 修改了库内或 `~/.claude` 下的命令/技能文件：**不会**自动更新，除非关闭/重开设置、切换会 `redisplay` 的选项，或在本页内再做一次增删改。

原因：无 Vault/`fs` 的 **watch**，列表是纯 **按需读取**。

## 3. 对话里斜杠「下拉」：何时更新（运行时 catalog）

这与设置页列表 **分离**。

### 3.1 入口

`TabManager` 为下拉配置的 `getEntries` 会先 `getSdkCommands(tab.id)`，再 `catalog.listDropdownEntries(...)`。

```659:670:src/features/chat/tabs/TabManager.ts
  private getProviderCatalogConfig(tab: TabData) {
    const providerId = getTabProviderId(tab, this.plugin);
    const catalog = ProviderWorkspaceRegistry.getCommandCatalog(providerId);
    if (!catalog) return null;

    return {
      config: catalog.getDropdownConfig(),
      getEntries: async () => {
        await this.getSdkCommands(tab.id);
        return catalog.listDropdownEntries({ includeBuiltIns: false });
      },
    };
  }
```

### 3.2 `getSdkCommands` 做了什么

从 **当前或兄弟标签** 已就绪的 `ChatRuntime` 取 `getSupportedCommands()`，并把结果写入 **同一个** `ClaudeCommandCatalog`：

```647:650:src/features/chat/tabs/TabManager.ts
    const catalog = ProviderWorkspaceRegistry.getCommandCatalog(providerId);
    if (catalog) {
      catalog.setRuntimeCommands(sdkCommands);
    }
```

### 3.3 `listDropdownEntries` 的策略（Claude）

- 若 catalog 里已有 **SDK 运行时命令**（或冷启动通过 **probe** 填充），则下拉以 **运行时列表** 为主（并过滤部分内置隐藏项）。
- 若 **空**，则 **回退** 到 `listVaultEntries()` 即 **文件扫描**（与设置页数据源一致）。

因此：**下拉**随「打开下拉/取条目」路径更新 SDK 缓存；**设置页**不经过 `getSdkCommands`，除非整页重绘触发重新 `listVaultEntries()`。

## 4. `ClaudeCommandCatalog.refresh()`

接口 `ProviderCommandCatalog` 定义了 `refresh()`。Claude 实现中为空操作（注释说明由外部 `setRuntimeCommands` 更新）。

```215:218:src/providers/claude/commands/ClaudeCommandCatalog.ts
  async refresh(): Promise<void> {
    // Claude revalidation happens externally via setRuntimeCommands
  }
```

Codex 侧技能 catalog 的 `refresh()` 则有实际 RPC/缓存逻辑，且与设置 UI 的刷新按钮配合（见下一节）。

## 5. 为何子代理、插件有「刷新」，命令与技能没有？

### 5.1 子代理 `AgentSettings`

头部有刷新按钮，点击调用 `agentManager.loadAgents()` 后 **整段 `render()`**。

```241:246:src/providers/claude/ui/AgentSettings.ts
    const refreshBtn = actionsEl.createEl('button', {
      cls: 'claudian-settings-action-btn',
      attr: { 'aria-label': t('common.refresh') },
    });
    setIcon(refreshBtn, 'refresh-cw');
    refreshBtn.addEventListener('click', () => { void this.refreshAgents(); });
```

**设计动机（推断）**：代理列表不仅来自 vault 文件，还与 **插件启用状态** 等发现逻辑相关，用户需要 **不离开设置页** 即重新 `loadAgents()`。

### 5.2 插件 `PluginSettingsManager`

刷新会 `pluginManager.loadPlugins()` 并联动 `agentManager.loadAgents()`，再 `render()`。

```137:148:src/providers/claude/ui/PluginSettingsManager.ts
  private async refreshPlugins() {
    try {
      await this.pluginManager.loadPlugins();
      await this.agentManager.loadAgents();

      new Notice(t('settings.pluginsUi.noticeListRefreshed'));
    } catch (err) {
      const message = err instanceof Error ? err.message : 'Unknown error';
      new Notice(t('settings.pluginsUi.noticeRefreshFailed', { message }));
    } finally {
      this.render();
    }
  }
```

**设计动机（推断）**：插件发现依赖磁盘/配置，与对话运行时分离，需要显式 **重新 discovery**。

### 5.3 Codex 技能 `CodexSkillSettings`

同时具备 **刷新按钮** 与 `catalog.refresh()` 调用，与 Claude 命令块 **UI 模式不一致**。

```176:199:src/providers/codex/ui/CodexSkillSettings.ts
  async refresh(): Promise<void> {
    await this.catalog.refresh();
    await this.render();
  }

  async render(): Promise<void> {
    this.containerEl.empty();
    ...
    const refreshBtn = actionsEl.createEl('button', {
      cls: 'claudian-settings-action-btn',
      attr: { 'aria-label': t('common.refresh') },
    });
    setIcon(refreshBtn, 'refresh-cw');
    refreshBtn.addEventListener('click', () => { void this.refresh(); });
```

### 5.4 Claude 命令与技能：无按钮 + `refresh()` 未接线的原因（归纳）

| 因素 | 说明 |
|------|------|
| **数据源** | 列表直接 **读文件**（加可选主目录），没有像插件那样的独立「发现服务」必须重跑。 |
| **历史产品形态** | 列表被认为在 **打开设置/重绘** 时已足够新；未对标 Codex 的「列表区统一带刷新」。 |
| **实现落差** | `SlashCommandSettings.refresh()` 已写但未挂载按钮、也未被 `ClaudianSettings` / `TabManager`  broadcasts 调用；与子代理/插件 **不对称**。 |
| **运行时与设置页分离** | 下拉靠 SDK；设置页靠 `listVaultEntries`，用户易认为两处应一致，但刷新路径 **本就不相同**。 |

## 6. 调查结论（给产品/研发的直接结论）

1. **设置页列表刷新**：构造时、页内编辑后、以及触发 **`redisplay`** 时；**没有**定时器与文件监听；**没有**刷新按钮；**存在未使用的** `refresh()` API。
2. **下拉刷新**：依赖 **`getSdkCommands` → `setRuntimeCommands`** 与 `listDropdownEntries` 的分支逻辑，与设置页 **非同一时机**。
3. **与子代理/插件的差异**：后两者刷新的是 **loadXxx 发现管线**；Claude 命令/技能以 **读文件** 为主， historically 未提供显式按钮，但在外部改文件场景下 **体验弱于**「一键重扫」。
4. **若需改进（非本报告范围，仅方向）**：可在 `SlashCommandSettings` 头部增加与 Codex 一致的刷新按钮并调用已有 `refresh()`；可选让 `ClaudeCommandCatalog.refresh()` 清掉 probe/runtime 缓存或强制重探（需单独方案）；长期可考虑对 `.claude/commands` / `skills` 做 **debounced** 的 vault 事件监听（需注意性能与跨平台）。
