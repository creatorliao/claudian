# C02 问题报告：斜杠 / 闪电按钮打开命令清单滞后

## 1. 背景与现象

在对话输入框中通过以下方式打开**斜杠命令与技能清单**时，用户感知到**明显延迟**：列表需等待一段时间才出现，影响「随手点开就选」的流畅度。

| 触发方式 | 说明 |
|---------|------|
| 键盘输入 `/` | 符合触发规则后应弹出下拉列表 |
| 导航栏 **闪电按钮** | 与手动输入 `/` 共用 `SlashCommandDropdown.openSlashPickerFromToolbar()`，同一套数据与逻辑 |

**期望**：列表应尽快呈现（至少先给出可交互的骨架或内置项），再按需补齐动态条目。

---

## 2. 根本原因（代码调查结论）

### 2.1 UI 层：展示被异步加载「拦住」

`SlashCommandDropdown.showDropdown()` 在渲染前先 **`await fetchProviderEntries()`**，只有异步完成后才执行 `buildItemList` → `render()`。因此**首次（或缓存失效后）打开**时，在下拉框出现在 DOM 之前必须等待整条异步链路返回。

相关代码：`src/shared/components/SlashCommandDropdown.ts` 中 `showDropdown`（约 238–265 行）与 `fetchProviderEntries`（约 267–280 行）。

### 2.2 数据层（Claude）：冷启动时常要走「SDK 探测」

对话 Tab 通过 `catalog.listDropdownEntries({ includeBuiltIns: false })` 拉取条目（见 `src/features/chat/tabs/Tab.ts` 中 `getRegistryProviderCatalogInfo`）。

`ClaudeCommandCatalog.listDropdownEntries`（`src/providers/claude/commands/ClaudeCommandCatalog.ts`）逻辑大意：

- 若运行时 **`sdkCommands` 仍为空** 且配置了 **`probe`**，会先 **`await ensureProbed()`**。
- `ensureProbed` 调用 **`probeRuntimeCommands`**（`src/providers/claude/commands/probeRuntimeCommands.ts`）：启动本地 **agent 会话**、等待 **`system/init`**，再调用 **`conversation.supportedCommands()`**，最后 abort。该路径涉及**子进程 / CLI / SDK 初始化**，耗时通常为**数百毫秒级甚至更久**，属于典型瓶颈。
- 若探测后仍无运行时条目，会 **`await listVaultEntries()`**：异步读取库内命令/技能文件，并在「库 + 用户主目录」范围内合并（多次 `loadAll`），同样有 I/O 延迟。

### 2.3 数据层（Codex）：列表依赖异步枚举技能

`CodexSkillCatalog.listDropdownEntries`（`src/providers/codex/commands/CodexSkillCatalog.ts`）需 **`await this.listProvider.listSkills()`**，再映射为下拉项；若后端枚举偏慢，同样会直接拉长 **`await fetchProviderEntries`**。

### 2.4 附带问题：空列表未计入下拉缓存

`SlashCommandDropdown.fetchProviderEntries` 仅在 **`entries.length > 0`** 时设置 `cachedProviderEntries` 与 `providerEntriesFetched`。若某次拉取结果为空，**每次打开仍会重新 await `getProviderEntries()`**，可能重复触发上述慢路径（需在优化方案中一并收紧）。

## 5. 改善方向（待你确认后再实施）

下列方向可按「收益 / 改动面」组合评估，不限于单选。

| 方向 | 思路 | 备注 |
|------|------|------|
| **A. 先渲染、后补齐** | 在完成 `getProviderEntries` **之前**先 `render()` 一层：例如仅内置命令、或占位「加载中」与骨架，数据返回后再增量更新列表 | 直接针对用户感知的「卡在空白」 |
| **B. 预热 / 后台拉取** | 在 Tab 就绪、会话恢复或插件空闲时**提前**调用 catalog 拉取或触发 probe，使首次打开时缓存已就绪 | 需注意与工作目录、会话切换时 `resetSdkSkillsCache` 的一致性 |
| **C. 缓存策略收紧** | 对「空数组」也标记已拉取（或短期 TTL），避免反复触发完整异步链路 | 小改动，可与 A/B 一起做 |
| **D. Claude 探测与列表职责拆分** | 评估是否能在不完全等同「完整 probe」的前提下展示静态/文件侧列表；probe 结果异步合并 | 需权衡与 SDK 行为一致性 |
| **E. 观测与验收指标** | 开发阶段对「从触发到首帧可见」打点（或手动计时），验收目标为相对优化前显著缩短（具体毫秒阈值待定） | 便于验证优化是否生效 |
