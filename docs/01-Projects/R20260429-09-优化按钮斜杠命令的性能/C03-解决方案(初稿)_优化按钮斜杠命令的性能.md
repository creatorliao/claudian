# C03 解决方案（初稿）：斜杠 / 闪电按钮命令清单响应优化

## 1. 目标

| 编号 | 目标 |
|------|------|
| G1 | 用户通过 **`/`** 或 **闪电按钮** 打开命令清单时，**首帧尽快出现**（明显优于当前「长时间无反馈」）。 |
| G2 | 列表内容**不与运行时/SDK 语义冲突**：仍沿用 **`getSdkCommands(tab)` → `catalog.listDropdownEntries`** 的顺序（见 `TabManager.getProviderCatalogConfig`）。 |
| G3 | 尽量避免 **「先出一截、再突然插入一批」** 的跳动；若做不到单次完整列表，则用可接受的过渡态（见 §3.3）。 |
| G4 | 不引入明显副作用：不必要的重复探测、错误 cwd、跨 Tab 串数据；并与现有 **工作区切换 / 会话切换** 缓存失效逻辑一致。 |

**依据**：同目录 `C02_问题报告_优化按钮斜杠命令的性能.md` 中的根因结论。

---

## 2. 策略总览（三层叠加）

采用 **「缓存收紧 + 后台预热 +（兜底）非阻塞首帧」** 组合；优先级从高到低实施与验收。

```
┌─────────────────────────────────────────────────────────────┐
│ ① 下拉组件：空列表也视为「已拉取」，避免重复慢路径              │
│ ② Tab/视图就绪后：后台调用与正式打开相同的 getEntries() 填缓存 │
│ ③ 若用户极快触发仍命中冷路径：首帧仅内置项或简短「加载中」再替换 │
└─────────────────────────────────────────────────────────────┘
```

- **①** 改动面小、风险低，单独即可消除「无命令却反复 probe」的浪费。
- **②** 是解决「丝滑」的主手段：首次点开前缓存已在内存。
- **③** 仅作为 **①② 仍不够快** 时的兜底，避免长时间空白；若 **②** 足够可靠，③ 可弱化或只做内置命令快显。

---

## 3. 方案详情

### 3.1 `SlashCommandDropdown`：缓存逻辑修正（必须）

**现状**：仅在 `entries.length > 0` 时设置 `providerEntriesFetched` 与 `cachedProviderEntries`，导致空结果每次打开都重新 `await getProviderEntries()`。

**调整**：

- 一次 `getProviderEntries()` **结束且无并发更新的请求**后，无论条目是否为空，均：
  - `cachedProviderEntries = entries`（可为 `[]`）
  - `providerEntriesFetched = true`
- `catch` 分支：可视为「已尝试」，同样标记已拉取并缓存空数组（或区分 `failed` 状态按需重试，初版建议 **记一次失败 + 空缓存**，避免无限重试；用户切换 Tab/会话时再 `reset`）。

**验收**：无任何自定义命令、探测返回空的场景下，连续多次打开下拉不应重复触发完整异步链路（可用日志或计数器验证）。

---

### 3.2 后台预热：与正式数据源完全一致（核心）

**原则**：预热必须调用 **与下拉相同的异步工厂**，即当前 Tab 上 **`getEntries`**（内部已是 `await getSdkCommands(tab.id)` 再 `catalog.listDropdownEntries(...)`），不得绕过 `getSdkCommands`。

**触发时机（建议全部采纳，实现时可合并为一次调度）**：

| 时机 | 说明 |
|------|------|
| **侧栏对话视图打开且 Tab 已创建** | `ClaudianView` / `TabManager` 在完成默认 Tab 创建或恢复后，对 **当前激活 Tab** 调度一次预热。 |
| **切换激活 Tab** | 对新激活 Tab 调度预热（可防抖，避免快速切换连跑多次）。 |
| **提供商切换** | 已有 `syncSlashCommandDropdownForProvider`，可在同步完成后对该 Tab 预热。 |

**落地方式（二选一或组合）**：

1. 在 `SlashCommandDropdown` 上新增 **`prefetch(): Promise<void>`**（或 package-private，由 Tab 调用）：内部等价于执行一次 `fetchProviderEntries` 的完整逻辑但不强制 `render()`；成功后与普通打开共用缓存字段。
2. 或在 `TabManager` 内抽出 **`warmSlashDropdownForTab(tabId)`**，内部取得该 Tab 的 `getProviderCatalogConfig(tab)`，调用 `await config.getEntries()`，再将结果写入对应 `tab.ui.slashCommandDropdown`（需下拉暴露「注入缓存」或与 `prefetch` 合并）。

**调度技巧**：

- 使用 **`requestIdleCallback`**（若环境不可用则 `setTimeout(0)`）推迟到首屏布局之后，减少对视图打开的阻塞。
- **同一 Tab 并发**：预热进行中用户又打开下拉 → 依赖现有 `requestId` 并发合并逻辑即可。

**成本说明**：会话未就绪时仍可能触发 probe（与当前首次打开一致）；预热只是把开销从「第一次点开」挪到「视图就绪后的空闲时段」，总体调用次数不增加（配合 §3.1 不回潮）。

---

### 3.3 兜底：仍冷路径时的首帧（可选）

若预热未完成用户已触发下拉：

- **优先**：若 §3.2 多数场景已足够快，可维持现有「等数据再 render」仅修正 §3.1。
- **可选增强**：`showDropdown` 在 **`providerEntriesFetched === false`** 时先 **`render()` 一层**：
  - **仅内置命令**（与现有 `includeBuiltIns` 规则一致），或
  - 单行文案「加载命令列表…」+ 禁用选择，待异步返回后 **一次性替换**整列表（避免逐项插入）。

**不推荐**：长时间显示不完整列表再逐项 `append`（易产生「中间插入」感）。

---

### 3.4 内联编辑（便签）路径

**现状**：内联编辑自建 `SlashCommandDropdown`，`getProviderEntries` 绑定 **inline 解析后的 catalog**。

**策略**：在 **内联编辑面板首次展示 / `resolvedProviderId` 就绪** 后，对当前实例调用与 §3.2 相同的 **`prefetch`**（或一次性 `getEntries()` 写入缓存）。不与侧栏共用 DOM，但 **同一 provider 的 `ClaudeCommandCatalog` 单例**仍可被两侧预热命中 catalog 层缓存（`setRuntimeCommands` 等）。

---

### 3.5 失效与现有逻辑对齐（必须复核）

以下路径已有或应有 **下拉缓存重置**，预热状态须一并失效或允许重新预热：

| 事件 | 现有行为（摘自仓库） | 预热侧要求 |
|------|----------------------|------------|
| 工作区路径切换 | `catalog.refresh()`、`slashCommandDropdown.resetSdkSkillsCache()` | 切换完成后对激活 Tab **重新调度一次预热**（idle）。 |
| 会话切换 / 加载 | `resetSdkSkillsCache` | 同上。 |
| 设置中 slash 来源范围变更 | 设置页会 `commandCatalog.refresh()` 等 | 关闭设置或保存后若视图仍打开，可对激活 Tab **预热**。 |

**不要求**：在外部文件监视器层面实时刷新列表（可作为后续增强）；本方案以「打开前预热 + 失效点重建」为主。

---

### 3.6 Codex 提供商

`getProviderCatalogConfig` 已对 Codex 走同一接口：`getEntries` 内含 `getSdkCommands` + `listDropdownEntries`。**无需单独一套预热分支**，只要预热调用 **`getEntries()`** 即可覆盖 Claude / Codex。

## 4. 验收标准（建议）

| 编号 | 检查项 |
|------|--------|
| AC1 | 侧栏视图打开后空闲片刻（例如 1～2 秒内无操作），再首次点击闪电或输入 `/`，**主观延迟**明显短于优化前（建议录制屏幕前后对比）。 |
| AC2 | 切换工作区子文件夹后，打开的清单与 **当前 cwd / 设置** 一致，无串 Tab、无过期 probe 长期使用。 |
| AC3 | **空命令库**场景：多次打开下拉不产生无意义的重复 probe（日志或计数）。 |
| AC4 | **回归**：斜杠选中插入、隐藏命令过滤、Esc 关闭等行为与优化前一致。 |

---

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 视图打开即触发 probe，用户从不使用斜杠 | 使用 **idle** 推迟；若需进一步省电可考虑「仅在有 CLI 路径或用户曾用过斜杠功能」再预热（可作为二期）。 |
| 预热与首次发送同时争抢 CPU | 接受短时叠加；预热放 idle 可降低冲突。 |
| 外部新增命令文件未及时出现 | 与现状一致；可选后续：设置页刷新、catalog 暴露「手动 refresh」时再预热。 |

3. 实测后决定是否启用 **§3.3**。
