# 解决方案（完善稿）：内联编辑路径与权限对齐

> 继承并替代初稿中的目标与技术要点；补充 **用户路径 ↔ 调用链** 对照、**产品先决条件**（含已落地门禁）、**遗漏与边界场景** 与 **验收清单**。  
> 调查细节仍以 `C01_调查报告与方案_内联编辑调用链与权限对齐.md` 为附录；**已落地的代码变更**见本文 **第 13 节** 与 C01 **第 10 节**。

## 1. 目标

1. **运行时路径对齐（Claude）**：内联构造 SDK `Options` 时与侧栏冷启动同源：经 `QueryOptionsBuilder.buildColdStartQueryOptions` 及与 `ClaudeChatRuntime.queryViaSDK` 一致的 `ColdStartQueryContext`，不再使用「`runColdStartQuery` + 固定 `bypassPermissions` + 只读工具/hook」的内联专用捷径。
2. **权限语义一致**：与侧栏相同的 `permissionMode`、安全模式映射、`canUseTool`（审批、AskUserQuestion、ExitPlanMode 等）、`disallowedTools` / MCP 策略；去掉内联人为的只读工具白名单与 `createReadOnlyHook`。
3. **编辑结果把关不变**：当前笔记中的落稿仍由内联 UI 的 Accept/Reject 与 `<replacement>` / `<insertion>` 解析应用；专用 `getInlineEditSystemPrompt()` 与用户消息结构不改（见第 4 节）。

## 2. 产品先决条件与用户心智（门禁已落地）

| 项 | 说明 |
|----|------|
| **必须符合才能使用内联** | 工作区内至少存在一个 `VIEW_TYPE_CLAUDIAN` 的叶子（`plugin.getView()`），且当前为 `MarkdownView` 上下文。 |
| **命令面板体验** | `editorCheckCallback` 在 `checking === true` 时若上述不成立，命令 **不出现/不可用**，避免误点；热键触发时 `checking === false` 仍会通过相同条件并 `Notice`。 |
| **「侧栏」的表述** | 实现上以 **存在 Claudian 叶子** 为准（可停靠在侧栏或主工作区）；不要求单独判断「侧栏分隔条是否展开」。 |
| **权限心智** | 用户理解为：**先有 Claudian，再在内联里改笔记**；审批与权限与当前聊天 Tab 绑定（见第 6 节边界场景）。 |

**代码**：`src/main.ts` 中 `claudian:inline-edit` 使用 **`editorCheckCallback`**（命令面板灰显）+ **`runInlineEditCommand`**（实际打开 `InlineEditModal`）。Claude SDK 路径与权限对齐仍见第 5、6 节（**待编码**）。

## 3. 用户操作路径 ↔ 程序调用链（模拟对照）

下列为 **主路径**（Claude、权限对齐 **实施后** 的目标链路；当前实现中 Claude 仍走 `runColdStartQuery`，以正文第 5 节为准）。

### 3.1 成功路径 A：打开聊天 → Markdown 中内联 → 完成

| 步骤 | 用户操作 | 程序行为（要点） |
|------|----------|------------------|
| 1 | 打开 Claudian（ ribbon / 命令） | 创建或显示含 `ClaudianView` 的叶子；`getView()` 非空。 |
| 2 | 在某一 Tab 中确保提供商与权限符合预期 | `setupServiceCallbacks` 已为 `tab.service` 绑定 `InputController` 审批等；`permissionMode` 读设置。 |
| 3 | 打开笔记，选中文本或仅放置光标 | 用户焦点在编辑器。 |
| 4 | 执行「内联编辑」命令/热键 | `main.ts`：`editorCheckCallback` → `runInlineEditCommand` → `InlineEditModal`。 |

| 步骤 | 用户操作 | 程序行为（要点） |
|------|----------|------------------|
| 5 | 在内联输入框输入指令并提交 | `InlineEditController.generate()` → `InlineEditService.editText` / `continueConversation`。 |
| 6 | （需审批时）在聊天区确认工具 | 目标：`canUseTool` 触达 **当前活跃 Tab** 的 `InputController`；用户需能看见侧栏/聊天区提示。 |
| 7 | 查看内联 diff，Accept / Reject | 与现版一致：`parseInlineEditResponse`、`apply` 逻辑不改。 |

**调用链（目标，Claude）**：

```text
editorCheckCallback (main.ts：门禁)
  → runInlineEditCommand
  → InlineEditModal.openAndWait()
    → InlineEditController（提供商解析仍依赖 getView().getActiveTab()…）
      → ProviderRegistry.createInlineEditService
        → ClaudeInlineEditService（目标）
          → runInlineEditAgentQuery / 等价封装
            → buildQueryOptionsContext（与 ClaudianService 同源）
            → QueryOptionsBuilder.buildColdStartQueryOptions
            → options.systemPrompt = getInlineEditSystemPrompt()
            → agentQuery({ prompt, options })
```

### 3.2 失败/拦截路径（当前与目标一致处）

| 步骤 | 用户操作 | 程序行为 |
|------|----------|----------|
| F1 | 未打开 Claudian 直接内联 | `Notice` `inlineEditRequiresClaudianView`，不打开模态。 |
| F2 | 非 Markdown / 无当前 MD 视图 | `Notice` `inlineEditNoMarkdownView`。 |
| F3 | 已有一实例内联，再触发一次 | 现版：`activeController.reject()`，第二次直接结束（避免并发）。 |

## 4. 内联特性与行为（实施中保持不变）

以下 **不** 随权限对齐重写（除非单独产品决策）。

| 类别 | 说明 |
|------|------|
| 命令入口 | `claudian:inline-edit`；`editorCheckCallback` 门禁（`getView()` + `MarkdownView`）+ `runInlineEditCommand`。 |
| 触发场景 | **必须** 存在 Claudian 视图（见第 2 节）；提供商与外部上下文仍优先来自 `getView()?.getActiveTab()`。 |

## 6. Claude 方案设计（实施要点）

### 6.1 路径与 systemPrompt

- 抽取 **仅内联使用** 的冷启动封装（示意名 `runInlineEditAgentQuery`），内部复用 `ClaudianService` 侧 `buildQueryOptionsContext` 的数据来源（或通过共享静态工厂避免重复拼 env/path/mcp/plugin）。
- 调用 `QueryOptionsBuilder.buildColdStartQueryOptions`，**最后覆盖** `options.systemPrompt = getInlineEditSystemPrompt()`（或在 builder 增加 `systemPromptOverride`）。
- **MCP**：与 `queryViaSDK` 一致地合并 `mcpMentions`、`enabledMcpServers`、`getActiveServers` / `getDisallowedMcpTools`；若内联消息或设置中带 MCP 提及，须传入同一套 `Set`，避免「侧栏能用 MCP、内联不能」。
- **`additionalDirectories`**：若侧栏冷启动传入 `externalContextPaths`，内联侧从「活跃 Tab / 设置」取同源列表（与 `queryViaSDK` 对齐）。
- **`enableFileCheckpointing`**：持久查询里为 `true`；冷启动 `buildColdStartQueryOptions` 当前未设置。若要对齐「文件检查点」语义，需在实施时对照 SDK 与侧栏行为**显式决定是否**为内联冷启动开启（避免与内联短会话假设冲突）——列为 **实施时二选一验证项**。

### 6.2 权限与 canUseTool

- 使用 **`createClaudeApprovalCallback`**，`getApprovalCallback` 等来自 **与 `InlineEditController` 解析的 `providerId` 对应的活跃 Tab 的 `tab.service`**（实施时需避免临时新建无回调的 `ClaudianService` 实例）。
- **门禁后仍可能出现的空回调**：视图已打开但 Tab 未就绪、`tab.service` 未 `setupServiceCallbacks`、或提供商非 Claude 时误走 Claude 内联服务——实施时需 **断言或降级**（Notice + 拒绝工具），避免静默 `deny` 无说明。
- **`allowedTools`**：侧栏冷启动可能在 `queryOptions.allowedTools` 收窄；内联默认不收窄（与常规对话 turn 一致），除非产品明确「内联禁止子代理」等。

### 6.3 其它 `runColdStartQuery` 调用方

- **标题生成等** 继续使用轻量 `runColdStartQuery`，**不** 改为重权限路径。

## 7. 遗漏与边界场景（综合检查清单）

实施与测试时逐项过一遍。

| # | 场景 | 期望/注意 |
|---|------|-----------|
| 1 | `getView()` 有值，但 `getActiveTab()` 为 null 或 `service` 未就绪 | 内联已打开 UI，提交时 `canUseTool` 是否可用；若无，应有明确 Notice。 |
| 2 | 多 Tab，活跃 Tab A 为 Codex，内联却解析成 Claude（或相反） | 核对 `InlineEditController` 提供商解析顺序与 **活跃 Tab** 一致；权限对齐后审批必须落在正确 `service`。 |
| 3 | 用户在编辑器内联，审批弹在聊天区 | 文档与用户提示：需要 **瞥一眼 Claudian**；可考虑后续「焦点闪烁」等体验增强（非本方案必选）。 |
| 4 | 计划模式 / ExitPlanMode / AskUserQuestion | `createClaudeApprovalCallback` 全链路必须在同一 `InputController` 上可跑通。 |
| 5 | `permissionMode` 动态切换 | 侧栏支持会话内更新；内联冷启动每次 `sendMessage` 应读 **当前** `plugin.settings`（与 `queryViaSDK` 一致）。 |
| 6 | 内联多轮 `continueConversation` | `resumeSessionId` 与首轮 **同一 Options 策略**；避免首轮对齐、续轮仍走旧 `runColdStartQuery`。 |
| 7 | 子代理 / Skill | hooks 与侧栏 `buildHooks()` 同源，避免内联侧缺 hook 导致行为分叉。 |
| 8 | 命令面板 / 热键在未打开 Claudian 时仍可选 | **已处理**：`editorCheckCallback` 在 `checking` 时返回 `false`，命令不显示；热键仍走 `checking === false` 分支并 `Notice`。 |
| 9 | 设置页「内联编辑」热键说明 | 可在 `settings.inlineEditHotkey` 描述中补充「需先打开 Claudian」（**文案增强**，非阻塞）。 |
| 10 | Codex/Cursor | 见第 9 节；避免假设「只对齐 Claude 即全产品一致」。 |

## 8. Codex / Cursor

- **Codex**：`CodexAuxQueryRunner` 中 `approvalPolicy: 'never'`、`sandbox: 'read-only'` 与 `CodexChatRuntime` 不一致；若要做到「内联 ≡ 侧栏」，需从 `resolveSandboxConfig()`（或等价）取策略并处理 **审批 UI 与 app-server** 的衔接。
- **Cursor**：`CursorAuxCliRunner` 已读全局 `permissionMode`；与 `CursorChatRuntime` 做一次 **参数级 diff** 回归即可。

## 10. 风险与约束

- 内联 **独立 SDK 会话** vs 侧栏持久会话：对齐的是 **权限与选项模型**，不要求 session id 合并（除非另起需求）。
- 审批 UI 在 **聊天视图**，内联在 **编辑器**：用户需适应「两处看」；门禁「先开 Claudian」降低预期错位。
