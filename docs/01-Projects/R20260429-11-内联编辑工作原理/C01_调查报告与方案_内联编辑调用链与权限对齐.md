# 内联编辑调用链、工作原理与权限对齐（调查报告与方案）

## 1. 背景与目标

内联编辑（在 Markdown 编辑器中通过命令触发的就地改写/插入）与侧边栏 Claudian 对话使用不同的运行时路径。当前 **Claude 提供商**下，内联路径在权限与可用工具上与侧边栏 **不一致**：例如侧边栏按用户在设置中选择的权限模式与审批流工作，内联侧则固定为「高绕过 + 只读工具集 + PreToolUse 拒绝写入类工具」。本文档梳理完整调用链、说明差异根因，并给出与侧边栏对齐的实施方案要点。

目标：在 **不改变内联编辑专属系统提示与 `<replacement>` / `<insertion>` 协议** 的前提下，使 **Claude（必要时应含 Codex）** 内联编辑的权限语义与侧边栏一致；Cursor 路径已部分复用全局 `permissionMode`，单独说明。

## 2. 内联编辑调用链路（从命令到 SDK/CLI）

### 2.1 入口：插件命令

用户在 Markdown/WYSIWYG 等场景绑定热键或命令 `claudian:inline-edit` 时，由 `main.ts` 注册 **`editorCheckCallback`**：`checking === true` 时若无 Claudian 或非当前 Markdown 上下文则命令在命令面板中不可用；`checking === false` 时校验同上，通过后委托 **`runInlineEditCommand`** 打开内联 UI。

0. **先决条件**：若工作区中不存在 Claudian 视图叶子（`plugin.getView()` 为空）或当前不是 `MarkdownView` 上下文，`editorCheckCallback` 返回 `false`：命令面板中该命令不可用；若通过热键等仍进入 `checking === false` 分支，则 `Notice`（`chat.notices.inlineEditRequiresClaudianView` 或 `inlineEditNoMarkdownView`）并返回。详见同目录 `C03-解决方案(完善稿)_内联编辑路径与权限对齐.md` 第 2、3、13 节。
1. 解析当前 `MarkdownView`；失败则提示。
2. 根据是否有选区，构造 `InlineEditContext`：`selection`（选中文本）或 `cursor`（含 `buildCursorContext` 的光标上下文）。
3. 实例化 `InlineEditModal`，传入 `editor`、`view`、`editContext`、`notePath`，以及用于拉取外部上下文的闭包（来自当前视图活跃 Tab 的 `externalContextSelector`）。
4. `await modal.openAndWait()`；若用户接受且返回了编辑正文，再出 Notice。

**代码位置**：`src/main.ts`（`id: "inline-edit"`：`editorCheckCallback`、`runInlineEditCommand`）。

### 2.2 UI 层：模态与控制器

`InlineEditModal.openAndWait()`：

1. 若已有全局 `activeController`，先 `reject` 避免并发。
2. 解析 `EditorView`（优先构造时传入的 `editor`，必要时回退 `view.editor`）。
3. `new InlineEditController(..., resolve)`，调用 `controller.show()`，以 Promise 等待用户 Accept/Reject。

`InlineEditController` 构造函数中 **解析当前聊天提供商**（用于选择哪个 `InlineEditService`）：

- 优先：`plugin.getView()?.getActiveTab()` 对应会话的 `conversation.providerId`；
- 否则：`activeTab.service?.providerId` 或 `activeTab.providerId`；
- 再否则：`DEFAULT_CHAT_PROVIDER_ID`。

随后：`this.inlineEditService = ProviderRegistry.createInlineEditService(plugin, providerId)`。

**代码位置**：`src/features/inline-edit/ui/InlineEditModal.ts`（`InlineEditModal`、`InlineEditController`）。

### 2.3 展示与输入

`show()` 会向 CodeMirror 挂载 `inlineEditField` 装饰、高亮选区；`createInputDOM()` 构建单行输入、斜杠命令下拉（`ProviderWorkspaceRegistry.getCommandCatalog(resolvedProviderId)`）、`VaultMentionDataProvider` 等。

用户提交后走 `generate()` → `inlineEditService.editText()` / `continueConversation()`。

### 2.4 提供商层：`InlineEditService` 实现

注册表按 `providerId` 返回：

| 提供商 | 实现类 | 底层执行方式 |
|--------|--------|----------------|
| `claude` | `ClaudeInlineEditService` | `@anthropic-ai/claude-agent-sdk` 的 `query`，经 `runColdStartQuery` |
| `codex` | `CodexInlineEditService` | `CodexAuxQueryRunner`（独立 app-server 线程） |
| `cursor` | `CursorInlineEditService` | `CursorAuxCliRunner`（Cursor CLI JSON 模式） |

**代码位置**：

- `src/core/providers/ProviderRegistry.ts`：`createInlineEditService`
- Claude：`src/providers/claude/auxiliary/ClaudeInlineEditService.ts`
- Codex：`src/providers/codex/auxiliary/CodexInlineEditService.ts`
- Cursor：`src/providers/cursor/auxiliary/CursorInlineEditService.ts`

### 2.5 提示词与响应解析

- `buildInlineEditPrompt(request)`：生成包含 `<editor_selection>` / `<editor_cursor>` 的用户消息；可附 `contextFiles`。
- `getInlineEditSystemPrompt()`：内联专用系统提示（与主对话 `buildSystemPrompt` 不同）。
- `parseInlineEditResponse`：从助手输出中提取 `<replacement>`、`<insertion>` 或澄清文本。

**代码位置**：`src/core/prompt/inlineEdit.ts`。

### 2.6 调用链小结（Claude）

```text
editorCheckCallback (main.ts：门禁) → runInlineEditCommand
  → InlineEditModal.openAndWait()
    → InlineEditController(show / generate)
      → ProviderRegistry.createInlineEditService(plugin, providerId)
        → ClaudeInlineEditService.editText | continueConversation
          → runColdStartQuery({ systemPrompt, tools, hooks, ... }, prompt)
            → agentQuery({ prompt, options })
```

## 3. 侧边栏对话（Claude）权限与工具链路（对比基线）

侧边栏单次发送通常走 `ClaudianService`（`ClaudeChatRuntime`）：持久会话或需冷启动时通过 `QueryOptionsBuilder.buildPersistentQueryOptions` / `buildColdStartQueryOptions` 构造 SDK `Options`。

要点：

1. **`applyPermissionMode`**：`permissionMode` 与 Claude **安全模式**（`safeMode` / `acceptEdits` 等）映射为 SDK 的 `PermissionMode`（如 `yolo` → `bypassPermissions`，`plan` → `plan`，其余走 `safeMode`）。
2. **`canUseTool`**：由 `createClaudeApprovalCallback` 提供，依赖 Tab 上通过 `setupServiceCallbacks` 挂载的 `approvalCallback`、`askUserQuestionCallback`、`exitPlanModeCallback` 等（实际 UI 在 `InputController`）。
3. **工具与限制**：默认全量可用工具减去 `disallowedTools`（含 MCP 禁用项、`UNSUPPORTED_SDK_TOOLS`、内置子代理黑名单等）；可选 `allowedTools` 收窄。
4. **Hooks**：`buildHooks()` 与主对话一致（与子代理、安全策略等一致），**不是**「仅只读工具 + 一律拒绝其它」的简化钩子。

**代码位置**：

- `src/providers/claude/runtime/ClaudeQueryOptionsBuilder.ts`：`applyPermissionMode`、`buildPersistentQueryOptions`、`buildColdStartQueryOptions`
- `src/providers/claude/runtime/ClaudeChatRuntime.ts`：`createApprovalCallback`、`queryViaSDK` 中组装 `ColdStartQueryContext`
- `src/providers/claude/runtime/ClaudeApprovalHandler.ts`：`createClaudeApprovalCallback`
- `src/features/chat/tabs/Tab.ts`：`setupServiceCallbacks`

## 4. 差异分析：内联编辑 vs 侧边栏（Claude）

### 4.1 `runColdStartQuery`（内联/标题等使用的简化路径）

`src/providers/claude/runtime/claudeColdStartQuery.ts` 中 **固定**：

- `permissionMode: 'bypassPermissions'`
- `allowDangerouslySkipPermissions: true`

不读取 `plugin.settings.permissionMode`，也不接收 `canUseTool`。标题生成等辅助任务同样走此路径（部分调用传 `tools: []`），与「主对话权限」无关是合理设计，但 **内联编辑若沿用此路径，则与用户期望的「和侧边栏同一套权限」冲突**。

### 4.2 `ClaudeInlineEditService` 额外约束

在 `sendMessage` 中：

- `tools: [...READ_ONLY_TOOLS]` —— 仅 `Read`、`Grep`、`Glob`、`LS`、`WebSearch`、`WebFetch`。
- `hooks.PreToolUse`：`createReadOnlyHook()` —— **非只读工具一律 deny**（含 `Write` / `Edit` / `Bash` / `Mcp` 等）。

因此即使 SDK 层为 `bypassPermissions`，模型也 **无法** 调用写入类工具；侧边栏在对应权限模式下却可以经 `canUseTool` 审批后执行。

### 4.3 与 `queryViaSDK` 冷启动路径的对比

`ClaudeChatRuntime.queryViaSDK` 使用 `QueryOptionsBuilder.buildColdStartQueryOptions`，其中：

- 权限来自 **用户设置** + `safeMode`；
- `canUseTool: this.createApprovalCallback()`；
- `disallowedTools`、`mcpServers` 等与对话上下文一致；
- 系统提示仍由 `buildBaseOptions` 中的 **主对话** `buildSystemPrompt` 生成——该路径服务于 **带编辑器块的主会话冷启动**，与内联的 **专用 system prompt** 不是同一条产品路径；对齐权限时 **不必** 混用主对话 system prompt，但 **应** 复用 `buildColdStartQueryOptions` 的权限/MCP/审批构造逻辑，并允许 **覆盖 `options.systemPrompt`** 为 `getInlineEditSystemPrompt()`。

### 4.4 对比表

| 维度 | 侧边栏 Claudian 对话（Claude） | 内联编辑（当前 Claude） |
|------|--------------------------------|------------------------|
| SDK `permissionMode` | 由 `permissionMode` + `safeMode` 解析 | 固定 `bypassPermissions` |
| `canUseTool` | 有（审批/AskUserQuestion/ExitPlanMode） | 无 |
| 工具白名单 | 默认全集 − `disallowedTools`（可选再收窄） | 仅 `READ_ONLY_TOOLS` |
| PreToolUse | 主对话 hooks | 额外只读钩子，拒绝非只读工具 |
| MCP | 随对话启用的 MCP 配置 | 未走 `buildColdStartQueryOptions` 的 MCP 合并逻辑 |
| 系统提示 | 主 Agent 提示 | 内联专用提示（应保持） |

## 5. 其他提供商（Codex / Cursor）

### 5.1 Codex

- 主对话：`CodexChatRuntime` 按设置解析 `approvalPolicy` 与 `sandbox`（如 `on-request` + `workspace-write` 等）。
- 内联：`CodexAuxQueryRunner` 在 `thread/start` 时 **写死** `approvalPolicy: 'never'`、`sandbox: 'read-only'`。

因此 Codex 下内联与侧边栏 **同样不一致**，且为 **沙箱 + 审批策略** 双层差异。

**代码位置**：`src/providers/codex/runtime/CodexAuxQueryRunner.ts`。

### 5.2 Cursor

- `CursorAuxCliRunner.query` 使用 `this.plugin.settings.permissionMode` 构建 `buildCursorAgentJsonModeFlagArgs`。

内联与标题等辅助查询 **已携带全局权限模式**，与「仅内联例外」的 Claude 路径不同；若仍有体感不一致，需单独对照 Cursor CLI 参数与主 `CursorChatRuntime` 是否 100% 同源（不在本文展开）。

**代码位置**：`src/providers/cursor/runtime/CursorAuxCliRunner.ts`。

## 6. 根因归纳

1. **历史拆分**：内联早期走独立 `runColdStartQuery`，追求实现简单、默认不让内联会话写盘；侧边栏走完整 `QueryOptionsBuilder` + 审批。
2. **安全意图与产品意图混淆**：「内联只返回 replacement XML」**不要求** SDK 侧永远只读——最终写回编辑器由用户 Accept 控制；与侧边栏对齐后，**仍可通过审批与权限模式** 约束危险工具。
3. **冷启动两条线**：`runColdStartQuery` 与 `buildColdStartQueryOptions` 在能力上不对等，内联未复用后者。

## 7. 解决方案（推荐分阶段）

### 7.1 Claude（必做，与需求直接对应）

**目标**：内联编辑的 SDK 选项在权限语义上与 **`QueryOptionsBuilder.buildColdStartQueryOptions` + `createClaudeApprovalCallback`** 一致，同时 **保留** `getInlineEditSystemPrompt()` 与当前 prompt 构建。

**建议实现要点**：

1. **新增或扩展冷启动入口**（避免破坏标题生成等仅需极简选项的调用方）  
   - 例如：新增 `runInlineEditColdStartQuery(config, prompt)`，或让 `runColdStartQuery` 支持可选 `mode: 'inline' | 'minimal'`，`inline` 分支从 `ClaudeChatRuntime` 或共享工厂组装 `ColdStartQueryContext`。
2. **组装与 `queryViaSDK` 同源的上下文**（不含持久会话 resume，或按需独立 `sessionId`）：  
   - `vaultPath`、`cliPath`、`effectiveCwd`、环境变量、`mcpManager`、`pluginManager`、用户 `settings`、`disallowedTools`、MCP mentions（若内联输入支持 @mcp，与对话一致）、`externalContextPaths`（若与附件/工作区一致）。
3. **`canUseTool`**：复用 `createClaudeApprovalCallback`，依赖与当前 **活跃 Tab** 相同的回调源。  
   - 在 `InlineEditController` 已有 `plugin.getView()?.getActiveTab()`；可约定：**内联编辑执行时以该 Tab 的 `tab.service`（Claude 时为 `ClaudianService`）暴露创建 approval 闭包**，或 **将 `InputController` 的 `handleApprovalRequest` 注入**（需注意提供者必须为 `claude` 且 service 已 `setupServiceCallbacks`）。  
   - **无可用 Tab / 无回调**：降级策略——拒绝工具并 Notice 提示「请打开 Claudian 侧栏并聚焦一个 Tab」，或回退到当前只读策略（产品决策）。
4. **移除** `ClaudeInlineEditService` 中针对内联的 `READ_ONLY_TOOLS` 固定白名单与 `createReadOnlyHook`，改为与冷启动对话相同的默认工具集（除非产品明确要求内联禁止 Bash/MCP，再作为 **额外** `disallowedTools` 配置项而非硬编码）。
5. **覆盖 system prompt**：在生成 `Options` 后设置 `options.systemPrompt = getInlineEditSystemPrompt()`（或等价参数），避免被 `buildBaseOptions` 的主对话提示覆盖——若当前 `buildColdStartQueryOptions` 未暴露 override，在 builder 中增加可选 `systemPromptOverride` 是最小侵入方式。
6. **会话**：内联多轮对话 `continueConversation` 使用 `resumeSessionId` 时，应仍在 **同一 Options 权限模型** 下续跑，避免首轮与后续轮策略不一致。

### 7.2 Codex（若需全提供商一致）

- 将 `CodexAuxQueryRunner` 的 `thread/start` 参数改为与 `CodexChatRuntime.resolveSandboxConfig()`（或同等函数）**同一套** `approvalPolicy` / `sandbox`（及必要时的 `sandboxPolicy`）。
- 评估 **read-only** 沙箱是否仍满足内联「只做读库辅助」场景；若用户侧栏为 `workspace-write`，内联应对齐，避免「侧栏可写、内联只能读」的困惑。

### 7.3 Cursor

- 以现状为基线做回归验证；若发现 CLI  flag 与主运行时仍不一致，再对照 `CursorChatRuntime` 与 `cursorLaunchArgs` 统一。

### 7.4 测试与验收

- 单元：权限模式枚举与 `resolveClaudeSdkPermissionMode` 映射不变；可对内联新入口抽 `ColdStartQueryContext` 做快照断言（mock mcp/plugin）。
- 集成/手工：  
  - 侧栏 `normal` + 需确认的工具 → 内联触发同类工具应弹出 **同一套** 审批 UI（`InputController`）。  
  - `yolo` / `plan` 与侧栏行为一致。  
  - 无侧栏 Tab 时的降级行为符合产品说明。

## 9. 结论

内联编辑与侧边栏权限不一致，**在 Claude 下是确定的设计分叉**：内联走 `runColdStartQuery`（固定 `bypassPermissions`）并叠加「只读工具 + PreToolUse 拒绝」，侧边栏走 `QueryOptionsBuilder` 与用户 `permissionMode`、`canUseTool`、完整 MCP/禁用工具逻辑。要将权限与侧边栏统一，应在 **保留内联专用 system prompt** 的前提下，让 Claude 内联复用 **`buildColdStartQueryOptions` 级别的选项构造**，并 **挂接与活跃 Tab 相同的审批回调**；Codex 需对齐 `CodexAuxQueryRunner` 的 `approvalPolicy`/`sandbox`；Cursor 已部分复用全局 `permissionMode`，可作为对齐参考。
