# C03 解决方案（初稿）· 配置界面精简与高级选项

**关联需求**：`C01_需求清单_配置界面精简与高级选项.md`  
**日期**：2026-04-29  
**状态**：初稿（待评审 / 实施后同步验收）

---

## 1. 方案概述

在**插件设置 Tab 的「通用」页**（实现上对应 `renderGeneralTab`）内增加一层 UI 分支：

1. 新增持久化字段，例如 `generalAdvancedOptionsEnabled: boolean`，**默认 `false`**（简易模式）。
2. 在通用页**顶部**渲染「显示更多选项」开关（文案与交互详见 **§3**）；关闭时仅渲染 **C01 3.2** 所列五项；开启时在开关下方渲染其余区块（可将现有代码拆分为「简易区块」与「高级区块」两个渲染函数，避免重复）。

**提供商子页**（Claude / Codex / Cursor 等）：初稿建议**第一阶段不改子页结构**，仅简化「通用」页；避免与 CLI、模型、MCP 等专业配置耦合在同一迭代。

---

## 2. 数据与默认值

| 项 | 策略 |
|----|------|
| `generalAdvancedOptionsEnabled`（命名可替换） | 默认 `false`；写入插件持久化设置（随 `saveSettings` 落盘）。 |
| `maxTabs`（最大聊天数） | **默认值改为 `5`**：默认设置常量中的 `maxTabs` 从 `3` 调整为 `5`（见 `src/app/settings/defaultSettings.ts`）；已安装用户在首次加载时若沿用旧 JSON，保持其已有数值，除非产品决定是否一次性迁移（建议：**仅新默认值作用于新安装或缺失字段回填**）。 |

## 3. 产品与文案规范（高优先级）

本节把「减轻恐惧感、降低漏配焦虑」落到可实施的文案与结构上；实施时以 **`zh-CN.json` / `en.json`** 为准，下列中文为**推荐稿**，英文需语义对等（勿硬译腔）。

**文案边界**：用户可见文案（含推荐稿、线框、占位符）**不出现产品或仓库品牌名**；智能体默认名称由程序决定，设置页仅用「默认助手名称」等泛称。英文可用 *the assistant*、*default display name* 等中性表述。本文技术段落尽量用「插件设置」「默认设置常量」等泛称，必要文件路径见小节 **§2**，避免在方案正文罗列含品牌的类型符号。

### 3.1 「高级 / 更多选项」开关：常驻与语义

| 要点 | 说明 |
|------|------|
| **位置** | 置于通用页**最顶部**（简易模式与展开高级后的完整模式均保留），避免用户产生「完整设置不见了」的恐慌；折叠的是**下方区块**，不是整张通用页。 |
| **心理模型** | **标题短、描述承担解释**：一句话交代「展开后大致有什么」，并明确 **「不必展开也能正常使用」**，缓解「不敢点」与「怕漏配」两种焦虑。 |
| **命名** | 优先采用 **「显示更多选项」**（中性、邀请探索）；若产品坚持用「高级选项」，则**必须**配套下文所述说明行，避免单独出现一个冷冰冰的「高级」。 |

**推荐文案（简体中文）**

| 元素 | 推荐稿 |
|------|--------|
| **标题（名称）** | `显示更多选项` |
| **说明（描述）** | `关闭时只显示常用项，足以日常使用。开启后可配置系统提示词、快捷键入口、环境变量等；不需要也能正常对话。` |

**推荐文案（English，语义对齐）**

| Element | Draft |
|---------|--------|
| **Title** | `Show more options` |
| **Description** | `When off, only common settings are shown—enough for everyday use. When on, you can configure system prompts, hotkey shortcuts, environment variables, and more. You don’t need this for normal chatting.` |

**开关为 ON 时（可选一行提示，置于开关下方、高级区块之上）**

- zh：`以下为补充设置，按需要调整即可。`
- en：`Optional settings below—adjust only if you need them.`

### 3.2 简易模式五项：微文案原则与推荐稿

目标：每条**一行说清「对你有什么用」**，少用内部术语；必要时在**描述**里写「默认 / 建议」，不把决策压力甩给用户。

| 配置项 | 原则 | 名称（如有调整） | 描述（推荐稿 · zh-CN） |
|--------|------|------------------|-------------------------|
| 用户名称 | 保持友好、可选感 | **名称推荐**：`希望如何称呼你？`（占位符与名称语义一致即可） | **描述推荐**：`用于问候与称呼；可不填。` |
| 智能体名称 | 与界面展示强关联 | 沿用「智能体名称」类表述即可 | **描述推荐**：`显示在对话界面左上角；留空则使用默认助手名称。`（默认名称由程序决定，界面文案不重复写死品牌名） |
| 聊天提供商 | **降低「全开才对」误解** | 分组标题可用「聊天提供商」或「启用的对话服务」 | **在分组下增加一行说明**（单独 small 文案或 `Setting` 描述）：`建议先只启用一个，熟悉后再按需打开更多。` |
| 语言 | 无门槛 | 沿用现有「语言」 | 沿用：`界面显示语言。` |
| 最大聊天数 | **避免「最大聊天数」孤立成抽象配额** | 优先使用类人话标题，或原标题 + 强描述二选一 | **名称推荐**：`同时打开的聊天标签数量`（或保留原名但在描述里解释）。 **描述推荐**：`侧边栏里可以同时保留的对话标签个数上限；越大占用越多空间。默认 5，一般够用。` |

英文环境下「最大聊天数」对应的标题建议：`Maximum concurrent chat tabs` 或 `How many chat tabs to keep open`，描述需包含 *default 5* 与 *upper limit* 的含义。

### 3.3 国际化 Key 建议（实施参考）

便于检索与评审，建议新增一组前缀清晰的键（名称可随仓库惯例微调）：

| 用途 | 建议 Key（示例） |
|------|-------------------|
| 开关标题 | `settings.moreOptions.name` |
| 开关描述 | `settings.moreOptions.desc` |
| 展开后提示（可选） | `settings.moreOptions.hintWhenExpanded` |
| 提供商简易说明 | `settings.providers.simpleModeHint` |
| 最大标签数标题（若与人话绑定） | `settings.maxTabs.simpleName` / `settings.maxTabs.simpleDesc` |

若采用「一套标题两套描述」，需在代码里根据「简易 / 高级」切换引用键，避免重复维护冲突。

## 4. 简易模式下的控件顺序与实现对应

与 `src/features/settings/` 下设置 Tab 实现中现有 `Setting` 对齐，建议顺序为：

1. **用户名称** — `settings.userName.*`
2. **智能体名称** — `settings.agentName.*`
3. **聊天提供商** — 现有 `renderProviderToggles(container)`（或等价抽层，保证仅在简易列表中出现一次）
4. **语言** — `settings.language.*`
5. **最大聊天数** — `settings.maxTabs`（滑块范围保持 3–10 等与现有一致；默认值为 5）

> 当前代码顺序为：语言 → 提供商 → … → 用户名称/智能体名称在中后部。**实施时需物理上移**「用户名称」「智能体名称」到通用页最前（仅在简易模式也适用；高级模式全开时顺序可按「用户期望」统一调整为：高级开关 → 简易五项 → 其余区块，以减少两套顺序的认知差异）。

## 7. 风险与待决

| 项目 | 说明 |
|------|------|
| 老用户 `maxTabs === 3` | 是否提示「已跟随默认改为 5」属产品决策；技术上可采用「仅缺省写入时 5」。 |
| 国际化 | §3 所列文案须落入 `zh-CN.json` / `en.json`；简易模式若使用独立标题/描述键，需与现有 `maxTabs` 等键并存或迁移策略一致。 |
| 重复渲染 | `renderGeneralTab` 较长，拆函数时注意 `this.display()` 刷新后开关状态与折叠区块一致。 |
| 文案长度 | Obsidian `Setting` 描述区不宜过长；若折行过多，可将提供商「建议先只启用一个」改为更小字号样式（需与现有设置页样式一致）。 |
