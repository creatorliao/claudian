## 1. 现象（与历史需求的区别）

- **R20260420-01 已解决：** 窗格内 `.view-content` 因主题 `padding` / flex 链导致的 **底缝**，输入区相对 **同一 leaf** 的底边对齐。
- **当前新问题：** 在多数主题下，**底部输入区与 Obsidian 全局「状态栏」在视觉上重叠**：状态栏叠在最上层，遮挡发送键、工具条等。观感高度约 **24～32px**，随字号与主题变化。

**与 v3 的叠加关系：** 壳层 **`padding-block: 0`** 让内容在 leaf 内 **贴底**；若状态栏对主工作区是 **叠层** 而非 **占位**，则「leaf 的底」低于「用户可点的安全底」，输入区会画进被遮挡区域。这与原需求文档 §4.3「状态栏鉴别」一致：需要在 **避让高度** 上单独处理，不能只靠「窗格内拉满」。

## 2. 官方变量与高度（无单一「标准 px」）

### 2.1 官方 CSS 变量

[Status bar · CSS variables](https://docs.obsidian.md/Reference/CSS+variables/Window/Status+bar)（[镜像](https://obsidian-developer-docs.pages.dev/Reference/CSS-variables/Window/Status-bar)）中与布局相关的要点：

| 变量 | 与本案的关系 |
|------|----------------|
| `--status-bar-font-size` | 影响实际行高与栏高 |
| **`--status-bar-position`** | 决定状态栏 **占位 vs 叠层** 的关键钩子 |
| **`--status-bar-scroll-padding`** | 可能与「为滚动内容预留底部」相关，**可优先读入 `calc`，数值因版本/主题而异** |
| 其它（背景、边框、圆角等） | 间接影响盒模型高度 |

文档 **未** 将 **`--status-bar-height`** 列为标准 Status bar 变量；社区代码里偶见 `var(--status-bar-height, 32px)`，应作 **可选项 + fallback**，勿强依赖。

### 2.2 本地如何量「真实栏高」

1. DevTools 选中 **`.app-container .status-bar`**（或实际根节点）。  
2. **Computed**：`height`、`padding`、`border`、`position`。  
3. 在 **`body` / `.app-container`** 上看 **`--status-bar-scroll-padding`**、**`--status-bar-position`** 的最终值。

## 3. Minimal 与其他主题：差异核对（能否在样式表上对齐？）

### 3.1 核心差异（布局语义）

| 维度 | Minimal（你反馈「完美」的常见情况） | 许多其它主题 / 默认叠层行为 |
|------|--------------------------------------|-----------------------------|
| **状态栏与文档流** | 在 **`minimal-status-off`** 等配置下，源码将 **`--status-bar-position: static`**，栏在 **流内占位** | 常保留 **与视口绑定的定位**，栏 **叠在工作区内容之上** |
| **工作区可用高度** | `.workspace` 有效高度 **已扣除** 状态栏占位 | 逻辑上可能仍「铺满」，**最底一行视觉落在栏下** |
| **Claudian v3 贴底** | 贴到的是 **真实内容安全区底**，一般不挡操作 | 贴到的是 **leaf 底**，易与 **叠层栏** 冲突 |
| **纯 CSS 能否「检测」模式** | 否：样式表 **无法** 可靠判断当前是 static 还是 fixed | 同上 |

Minimal 出处（节选，便于对照）：[`kepano/obsidian-minimal` · `src/css/main.css`](https://github.com/kepano/obsidian-minimal/blob/master/src/css/main.css)

```css
.minimal-status-off .status-bar {
  --status-bar-position: static;
  /* … 边框与背景等 … */
}
```

> **说明：** 若你 **未** 开启 `minimal-status-off` 仍觉得正常，可能是该环境下栏高较矮、对比度弱、或与其它插件/片段组合后恰好未挡关键按钮；**排查仍以 DevTools 的 `position` 为准**。

### 3.2 「我们能不能在样式表上做一些调整？」—— 可以，但分两层

| 层级 | 是否建议由 Claudian 做 | 做法 |
|------|-------------------------|------|
| **A. 仅 Claudian 窗格**（**推荐默认**） | **是** | 在现有 **`obsidian-layout-claudian-view.css`** 中，保持选择器 **`[data-type="claudian-view"]`**，对 **`.view-content.claudian-container`**（或已约定的壳层）增加 **`padding-bottom`** / 自定义变量，用 **`calc` + 官方变量 + 保守 fallback**（见 §4）。**不** 修改全局 `.status-bar`。 |
| **B. 全局状态栏** | **否（插件默认）** | 例如全局 `.status-bar { --status-bar-position: static }` 会改变 **整个 Vault** 的 UI 契约，易与主题冲突；仅适合用户 **自备 CSS 片段**（§4.5）。 |

**结论：** 可以在 **插件自带样式表** 里 **只对 Claudian 视图** 做底部避让，从而在 **叠层主题** 上接近 Minimal「不被挡」的体验；**无法** 用一张样式表让 Obsidian 所有主题的状态栏行为与 Minimal 完全一致，除非采用全局改写（不推荐作为插件默认）。

## 4. 推荐实施方案（待你确认后落盘代码）

### 4.1 原则

- 保留 v3：**`padding-inline`** 左右留白不变。  
- **新增** 底部避让，目标：叠层主题下输入区 **完全可点**；流内主题下 **尽量避免「双倍空白」**（见 §4.3）。  
- 暴露 **`--claudian-view-bottom-safe`**（及可选 **`--claudian-view-bottom-safe-extra`**），便于用户片段覆盖。

### 4.2 推荐变量定义（方向性，非最终定稿）

在 **`.workspace-leaf-content[data-type="claudian-view"]`** 上定义（与现有 `--claudian-view-*` 同级），例如：

```css
.workspace-leaf-content[data-type="claudian-view"] {
  /* 待确认数值：fallback 建议用你在「问题主题」下量的 .status-bar 高度，常见 28～32 */
  --claudian-view-bottom-safe: max(
    var(--status-bar-scroll-padding, 0px),
    var(--status-bar-height, 0px),
    28px
  );
}
```

对 **`.view-content.claudian-container`**：

```css
padding-bottom: var(--claudian-view-bottom-safe) !important;
```

**说明：**

- **`max(...)`**：多种来源取较大者，减少「变量有值但偏小」时的遮挡；若某主题 **`--status-bar-scroll-padding` 已很大**，会自动抬高，一般仍安全。  
- **`var(--status-bar-height, 0px)`**：无定义时为 0，不报错；若运行环境有定义则参与 `max`。  
- **最终 `28px`** 仅为占位示例，**应以你复现主题下的实测为准**（可改为 `30px` / `32px`）。

### 4.3 「流内主题」与双倍空白（Minimal 类）

- **成因：** 状态栏已 **占位**，工作区底已在栏上沿；若再对 `.view-content` 加一整段 **`padding-bottom`**，会在 **leaf 内部** 多出一条「空白带」，**不** 一定与状态栏「叠两次」，但 **观感** 可能像底部过空。  
- **纯 CSS 无法自动关掉避让。** 实用策略：  
  - 文档说明：在 Minimal / 已 static 的用户可将 **`--claudian-view-bottom-safe: 0`**（或 `4px` 仅作呼吸缝）；  
  - 或在实现定稿时评估：**默认 fallback 略保守**（宁可略空少挡），由反馈再调。

### 4.4 加在哪一层 DOM？

| 目标元素 | 效果 |
|----------|------|
| **`.view-content.claudian-container`**（**推荐**） | 消息区 + 输入区整体上移，**一致性好**。 |
| 仅 **`.claudian-input-container`** | 只抬输入区，消息列表底可能仍贴边，**一般不推荐** 作为唯一手段。 |

### 4.5 移动端（可选）

若需与桌面同文件：**同一变量内** 增加 `env(safe-area-inset-bottom, 0px)`（再与上述 `max` 组合），避免刘海屏贴底裁切。是否纳入以产品范围为准。

### 4.6 用户侧（非插件）

- **验证假设：** 仅对 Claudian 的 `.view-content.claudian-container` 试 `padding-bottom: 32px !important`。  
- **全局模仿 Minimal（高级）：** 用户片段中 `.status-bar { --status-bar-position: static; }` —— **插件默认不做**（§3.2）。

| 步骤 | 内容 |
|------|------|
| 1 | 至少 **3 套主题**：**Minimal**、Obsidian **默认**、**问题主题**（如 Kanagawa Paper / Notation 等你截图中的）。 |

| 3 | 验证 **发送按钮、底栏图标** 可点、无遮挡。 |
| 4 | **Minimal**：若底部空白过大，测 **`--claudian-view-bottom-safe: 0`** 是否可接受。 |
| 5 | **分屏**：Claudian 与邻窗底边观感可接受即可（略上移为预期代价）。 |
