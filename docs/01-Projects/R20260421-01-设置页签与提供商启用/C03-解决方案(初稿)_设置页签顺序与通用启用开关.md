# C03 解决方案：设置页签顺序与通用「提供商启用」入口

> [!note] 与初稿的差异
> 下述 **已实施**（C06）。原「待确认」中的 **(1) 三提供商全关** 已定为 **不允许**，由 UI 拦截；(2) **已有 Tab 使用某提供商时被关闭**：仍依赖现有 `saveSettings()` 内的 **`ProviderSettingsCoordinator.normalizeProviderSelection`**，与初稿倾向一致。

## 2. 原现状摘要（实施前，留档）

- **注册顺序**（`src/providers/index.ts`）：`claude` → `codex` → `cursor`。设置子页签曾依赖键顺序，与 R1 不一致。
- **`blankTabOrder`**：曾导致 **空白 Tab / 混合模型列表** 分组顺序与「Claude → Cursor → Codex」不一致。
- **Claude** 曾为 `isEnabled: () => true`，无 `enabled` 字段。
- 设置 UI 曾展示 **全部** 注册提供商子页签；Codex/Cursor 子页内另有「启用」Toggle。

---

## 3. 方案原则（不变）

- **单一事实来源**：各提供商 `enabled` 仅存于 `providerConfigs.<id>`。
- **显式顺序**：`PROVIDER_UI_ORDER`（`src/core/providers/providerUiOrder.ts`）统一 **Claude → Cursor → Codex**，供设置页签与 `getEnabledProviderIds` 排序使用。

---

## 4. 数据与配置（落地）

### 4.1 Claude `enabled`

- `ClaudeProviderSettings.enabled`，默认 **`true`**；`claudeProviderRegistration.isEnabled` 读取该字段。

### 4.2 `getEnabledProviderIds` 排序

- `ProviderRegistry.getEnabledProviderIds` 对已启用 id **按 `getProviderUiRank` / `PROVIDER_UI_ORDER` 排序**（不再依赖 `blankTabOrder` 相减）。
- 各提供商 `blankTabOrder` 已调整为 **Claude 8、Cursor 10、Codex 12**，与产品顺序一致（若其它代码仍读该字段）。

### 4.3 「不允许三提供商全关」（产品决策 · 已锁）

- **不允许** 三个 `enabled` 同时为 `false`。
- **实现位置**：`ClaudianSettings.renderProviderToggles`：用户将某一 Toggle 置为 **关** 后，若 `getEnabledProviderIds` 为空，则 **回滚该次写入**、恢复 Toggle 状态，并 **`Notice(t('settings.providers.atLeastOneEnabled'))`**。
- 因此 **不要求** 在 `resolveSettingsProviderId` 上单独实现「零启用」分支；运行时仍以至少一个启用为前提。

---

## 5. 边界：开启时检测本机 CLI（已落地）

### 5.1 目标

用户将某提供商从 **关 → 开** 时，若按 **与运行时相同** 的路径解析逻辑 **无法找到对应 CLI**，则 **不保存为启用**，恢复 Toggle，并 **`Notice` 提示**（引导到各提供商页签配置路径或安装 CLI）。

### 5.2 实现模块

| 项目 | 说明 |
|------|------|
| 辅助方法 | `src/features/settings/providerCliPresence.ts` — `isProviderCliPresent(providerId, settingsBag)` |
| Claude | `resolveClaudeCliPath(hostname, legacy, env)`（与 `ClaudeCliResolver` 一致） |
| Cursor | `resolveCursorCliPath`（`CursorBinaryLocator`） |
| Codex | `resolveCodexCliPath`（`CodexBinaryLocator`）；另见 5.3 |

### 5.3 Codex + Windows WSL

- **宿主侧** 不一定存在 `codex.exe`；可执行文件在 WSL Linux 环境内。
- **处理**：`process.platform === 'win32'` 且 `installationMethod === 'wsl'` 时，`isProviderCliPresent('codex', …)` **直接返回 `true`**，**不**在 Windows 上因找不到文件而拦截启用，避免误报。
- 单测：`tests/unit/features/settings/providerCliPresence.test.ts` 覆盖该分支。

### 5.4 国际化

- `settings.providers.cliMissing.claude` / `.cursor` / `.codex`（`en.json`、`zh-CN.json`，键已登记于 `src/i18n/types.ts`）。

## 6. UI 与调用链（落地摘要）

- **通用**：语言等之后、「显示」等之前，增加 **「聊天提供商」** 小节（`settings.providers.heading`）及三个 Toggle（`settings.providers.toggle.*`）。
- **子页签**：`'general' + PROVIDER_UI_ORDER 中且 isEnabled 为真的提供商`；`activeTab` 若指向已隐藏子页签则回到 `general`。
- **CodexSettingsTab / CursorSettingsTab**：已移除子页内「启用提供商」重复项。
- **保存后**：`ProviderSettingsCoordinator.reconcileTitleGenerationModelSelection`（若需二次 `saveSettings`）、各视图 `refreshModelSelector`、`display()` 重绘设置页。

## 8. 风险与缓解（更新）

| 风险 | 缓解 |
|------|------|
| 全关导致 `settingsProvider` 脱节 | **UI 禁止全关**，不从数据层依赖「零启用」 |
| 无 CLI 仍强行启用 | **开关路径**：关→开时 `isProviderCliPresent` 拦截 + Notice |
| WSL Codex 误报未安装 | **WSL 分支** 宿主不校验可执行文件路径 |
