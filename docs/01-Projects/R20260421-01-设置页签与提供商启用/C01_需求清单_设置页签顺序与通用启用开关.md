# C01 需求清单：设置页签顺序与通用「提供商启用」入口

## 元信息

| 项 | 内容 |
|----|------|
| 录入日期 | 2026-04-21 |
| 主题关键词 | 设置、页签顺序、提供商启用、模型下拉、去重 |
| 状态 | 待方案设计（C03） |

## 业务价值

- **认知负荷**：把「是否使用某个 Agent/提供商」集中在「通用」一页决策，避免在多个子页签重复找「启用」开关。
- **导航清晰**：顶栏页签顺序与「主产品 → 扩展」的心智模型一致（先 Claude，再 Cursor，最后 Codex）。
- **一致性**：关闭某提供商后，界面不再出现无用入口（页签、模型），减少误选与困惑。

## 用户场景

1. 用户只使用 Claude，希望关掉 Cursor/Codex，不在设置顶栏和模型列表里看到它们。
2. 用户希望在「通用」里一次性打开/关闭三个提供商，再到各提供商页签只调路径、环境等细节。
3. 用户厌恶 Codex、Cursor 页签里再放一个与通用重复的「启用提供商」开关。

## 需求描述

### R1：设置页签从左到右顺序

- **期望顺序**：`通用` → `Claude` → `Cursor` → `Codex`（从左往右）。
- **说明**：与当前实现中由 `ProviderRegistry` 注册顺序或 `blankTabOrder` 等决定的顺序可能不同，需按上述固定或显式排序，避免随注册顺序变化。

### R2：通用页签中的三个 Toggle（提供商开关）

- 在 **通用（General）** 页签中增加 **三个** 开关，分别控制三个 Agent/提供商是否启用：**Claude**、**Cursor**、**Codex**。
- **当某一提供商被关闭时**：
  - 设置界面 **顶部不应再出现** 该提供商对应的 **子页签**；
  - 聊天相关 UI（至少 **模型选择下拉**）中 **不应再出现** 该提供商下的模型或选项（与「未启用」语义一致）。
- **当重新打开** 时：恢复页签与模型可见性（具体是否需刷新视图可在方案中约定）。

### R3：子页签内不再重复「启用提供商」

- **Codex**、**Cursor**（以及若 Claude 将来存在同类开关）**提供商子页签** 中，**不应再重复** 提供与 R2 等价的「启用该提供商」类控件与说明。
- 子页签仅保留该提供商的 **路径、环境、安全说明、Skills** 等 **非「是否启用」** 的配置。
- 若需提示「当前提供商未启用」，可用简短只读提示 + 引导回「通用」打开（可选，方案阶段定）。

## 现状与实现提示（供 C03 方案引用）

- **Cursor / Codex**：子设置页已有「启用提供商」类 Toggle（如 `settings.codex.providerEnabled`、`settings.cursor.providerEnabled`），与 R2/R3 直接相关——迁移到通用后应 **移除子页重复项** 并 **绑定同一套设置字段**（或合并为统一结构，由方案定）。
- **Claude**：当前注册为 `isEnabled: () => true`（恒启用），**尚无** 与用户可切换的「关闭 Claude 提供商」等价的设置项。若产品坚持「三开关对齐」，需在方案中 **新增 Claude 的启用状态**（默认值、迁移、与现有会话兼容性）。
- **页签与模型过滤**：`ClaudianSettings` 中 Tab 列表与 `ProviderRegistry.getEnabledProviderIds` / 模型选择器逻辑需与 R2 一并梳理，避免漏网（例如 `settingsProvider` 指向已禁用提供商时的回退行为）。

## 完成标准（验收口径）

- [ ] 设置顶栏页签顺序为：通用 → Claude → Cursor → Codex（在无本地化/主题干扰下目视与 DOM 顺序一致）。
- [ ] 通用页存在三个独立 Toggle，能分别控制三个提供商的启用状态；状态持久化。
- [ ] 任一提供商关闭后：对应子页签不显示；模型下拉中不可选到该提供商模型（或等价不可见行为）。
- [ ] Codex/Cursor 子页签中不再出现重复的「启用提供商」主开关（文案与控件均无重复）；Claude 子页若无同类项则保持不新增重复项。
- [ ] 回归：启用全部时行为与改版前一致或经方案确认的可接受差异（会话、默认模型、安全模式等）。
