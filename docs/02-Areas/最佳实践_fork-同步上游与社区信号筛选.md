# 最佳实践：Fork 同步上游与社区信号筛选

本文面向：**在官方开源仓库（上游）基础上维护自己 fork** 的维护者。目标不是「把上游所有东西都搬过来」，而是**只把本仓库还没有、且对自己有价值的东西**合进来，并**用社区 Issue/PR 作为风险与机会的信号**，而不是被信息量淹没。

---

## 1. 你真正需要的是什么（本质诉求）

可以概括为三件事：

| 诉求 | 含义 |
|------|------|
| **补缺口** | 上游 `main` 已合并、但**本 fork 历史里还没有**的提交（或等价修复）。这是「拉取」的核心对象。 |
| **看风险** | 开源项目里**大量用户**在 Issue 里报的错误、环境、复现路径——用来预判自己会不会踩坑、以及**该不该优先对齐某次上游修复**。 |
| **选机会** | 上游仍 **Open** 的 PR 里，哪些**功能/修复**与你的产品路线一致、值得合并或 cherry-pick；哪些**看看即可**。

反过来说：**不需要**把「上游仓库在两周内发生的所有新闻」都当成自己的待办；**不需要**把「别人开的每一个 PR」都合并进来。

---

## 2. 真正需要从上游「拉」的是什么

### 2.1 必拉：代码缺口（提交级）

- **对象**：`upstream/main` 上存在、但本分支 `HEAD` 不包含的提交（尤其是**带 PR 合并说明的修复**）。
- **核对方式**（在仓库根目录）：

```bash
git fetch upstream
git merge-base HEAD upstream/main
git log --oneline HEAD..upstream/main
```

- **含义**：`HEAD..upstream/main` 列出的**每一条**都是「你还没合进来的上游提交」。**这才是「拉取」时真正需要优先处理的代码**。

若列表为空，说明在**共同祖先**意义上，你已不落后于上游（或仅文档/发布节奏不同）；**不必**为了「同步感」而强行合并。

### 2.2 建议拉：与本 fork 无冲突的「安全合并」

- **merge upstream/main** 或 **rebase** 到本 `main`：在**解决冲突**后，一次性带上所有缺口提交。
- **cherry-pick 单个提交**：当你只想拿某一个修复、不想整包合并时（例如先验证一个 PR 的补丁）。

### 2.3 可选拉：Release 与标签

- 若你用 **GitHub Releases** 或 **manifest 版本** 对齐用户预期，可关注上游 **Latest** 与 **CHANGELOG**，但**版本号不必与上游一致**（fork 常有独立版本线）。

---

## 3. 哪些「不必拉」或「不必追着合并」

| 类型 | 说明 |
|------|------|
| **已在共同祖先里的上游历史** | 若 `merge-base` 已包含某次 PR 合并，则该 PR **不再**是你的「待拉清单」项。 |
| **本 fork 已用别的 PR 实现的功能** | 例如上游 Open **#530**，你本仓库已合并 **#523** 同功能线——**不必**再为「上游 PR 号不同」重复合并，除非上游后续有**额外修复**，再对比 diff。 |
| **纯文档/纯调查** | 上游的 README、他人写的调查报告、与运行时代码无关的仓库笔记——**按需** cherry-pick 或手写，**不**自动进合并流程。 |
| **大体量、多提供商、长期维护成本高的 PR** | 例如「再接入一种全新 Agent/协议」：除非明确需要，否则**先观望**，等上游主线稳定或社区共识收敛。 |
| **实验分支** | 未进 `main` 的 feature 分支、长期 Open 且无维护者合并意向的 PR——**默认不拉**，除非你愿意承担维护成本。 |

---

## 4. 如何用 Issue 与 PR：「社区错误」与「值得合并」

### 4.1 Issue：看「别人遇到什么错误」

- **目的**：了解**高频环境**（Windows、WSL、Codex、某 CLI 版本）、**回归**、**与上游某次修复的对应关系**。
- **用法**：
  - 按 **label**、**最近更新**、**reaction** 排序，优先看「多人复现」「有明确版本号」的。
  - 对照 **`git log HEAD..upstream/main`**：若某 Issue 写明「已在 PR #xxx 修复」，先确认 **#xxx 是否已在你 fork 里**（见第 2 节），再决定是**合并上游**还是**本地仍复现**。

**不要**把「上游已关闭的 Issue」自动当成「我这边也好了」——**必须**以**本仓库是否包含对应提交**为准。

### 4.2 PR：哪些「值得合并」

建议在合并前快速过一遍：

| 维度 | 值得考虑合并 | 建议暂缓 |
|------|----------------|----------|
| **与缺口一致** | 修复了 `HEAD..upstream/main` 里相关问题的 Open PR（或上游合并后你缺的提交） | 与当前缺口无关的大功能 |
| **与风险一致** | 对应 Issue 里你**正在复现**或**用户会复现**的路径 | 仅少数边缘环境、你无法测试 |
| **与路线一致** | 增强你 fork 已有的能力（例如上下文、输入体验、某提供商稳定性） | 再接入一整条新提供商/新协议 |
| **可维护性** | 改动面清晰、有测试、与现有架构一致 | 巨型 diff、与本地大量改动冲突 |

**合并策略**：优先 **跟随上游 `main` 已合并项**（风险低）；对 **Open PR**，优先 **cherry-pick 或等作者合并进上游后再拉**，避免长期与上游分叉。

---

## 5. 推荐节奏（可执行的小流程）

1. **固定频率**（如每周或每两周）：`git fetch upstream` → 看 `git log HEAD..upstream/main` 条数与内容。  
2. **有缺口**：合并或 cherry-pick → 跑 `typecheck` / `lint` / `test` / `build`（按项目约定）。  
3. **并行**：浏览上游 **Open Issues** 中最近高互动条目，**只记录**与本环境/路线相关的编号，写入你自己的项目笔记（不必合并进仓库）。  
4. **Open PR**：维护一张**短清单**（仅 PR 号 + 标题 + 是否已在本 fork 有等价物），**每轮同步**时勾选或删除。  

---

## 6. 与本仓库文档的衔接

- 周期性「上游两周调查」类文档：适合**存档**与**沟通**，但**真正操作**仍应以 **第 2 节 Git 命令** 与 **合并后的本地验证**为准。  
- 调查报告里的「待合并 PR」**必须**随 `merge-base` 更新而**删减**（已合并进本 fork 的不再出现），避免重复劳动。

---

## 7. 一页检查清单（合并前自问）

- [ ] `HEAD..upstream/main` 是否为空？非空则**优先处理**这些提交。  
- [ ] 本次要合并的 PR/提交，是否已在**本 fork 历史**或**等价实现**里？  
- [ ] 对应 Issue 是否仍适用于**当前** `main`？  
- [ ] 合并后能否在**本机目标环境**（至少一种）跑通构建与核心路径？  
- [ ] 大体量 PR 是否值得**长期维护成本**（新提供商、新协议）？  

---

## 8. 小结

**拉取**的本质是：**把上游 `main` 里你还没有的提交，按需要合进来**；**不是**订阅上游仓库的全部活动。  
**Issue / PR** 的本质是：**用社区信号做风险与机会管理**；**不是**把每一条都变成自己的合并任务。

按此实践，你可以在**保持 fork 独立演进**的同时，**可控地**吸收上游修复与有价值的社区贡献。
