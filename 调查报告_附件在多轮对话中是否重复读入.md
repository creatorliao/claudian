# 调查报告：附件常驻输入区时，多轮发送是否会重复读入文档

**调查范围**：Claudian 插件（Obsidian）内「文件上下文」与发送逻辑；并补充对多轮 LLM 会话的一般性说明。  
**结论摘要**：**拖拽附加的文件路径不会在每一轮用户消息里重复注入 `<context_files>`**；界面上的附件 chip 可以一直显示，但**从第二次发送起，插件侧不会再把同一份「待发送拖拽队列」里的路径打包进新的一条请求**。文档「是否算重复进入模型上下文」要区分：**路径元数据**（可能随历史保留）与**正文全文**（通常只在首轮由模型通过工具或历史中的助手回复进入上下文）。

---

## 1. 用户场景复述

- 一份文档以「附件 / 上下文 chip」形式留在输入区附近。
- 第 1 次发送时，模型已经结合该文档处理过问题。
- **问题**：第 2、第 3 次继续发消息时，系统会不会**再读一遍**这份文档、会不会**再发一遍**相同内容？

---

## 2. Claudian 代码层面的行为（事实依据）

### 2.1 拖拽文件：发送时「一次性消费」队列

拖拽进聊天的库内文件会进入 `pendingContextFiles`；**每次构建发送请求**时调用 `consumeContextFiles()`：**取出当前队列中的路径并立刻清空队列**。

```656:659:src/features/chat/controllers/InputController.ts
    // 拖拽附加的文件在发送时一次性消费，避免重复注入到后续轮次
    const contextFiles = !isCompact
      ? (fileContextManager?.consumeContextFiles() ?? [])
      : [];
```

```187:191:src/features/chat/ui/FileContext.ts
  consumeContextFiles(): string[] {
    const files = Array.from(this.pendingContextFiles);
    this.pendingContextFiles.clear();
    return files;
  }
```

因此：

| 轮次 | `contextFiles` 是否非空（拖拽场景） | 是否在新消息末尾追加 `<context_files>…` |
|------|--------------------------------------|----------------------------------------|
| 第 1 次发送 | 是（若本轮前有拖拽且未发过） | 是（由编码器追加） |
| 第 2 次及以后 | 否（队列已空，除非用户再次拖拽） | 否 |

编码器仅在 `request.contextFiles` 有内容时才追加 XML 块：

```24:26:src/providers/claude/prompt/ClaudeTurnEncoder.ts
    if (request.contextFiles && request.contextFiles.length > 0) {
      persistedContent = appendContextFiles(persistedContent, request.contextFiles);
    }
```

Codex 路径同理，使用 `[Context files: …]` 形式，同样依赖 `request.contextFiles`（见 `encodeCodexTurn.ts`）。

**含义**：插件**刻意避免**在后续轮次重复注入同一批拖拽路径；注释已写明设计意图。

### 2.2 界面 chip 与队列是两套状态

附件在界面上的展示依赖 `FileContextState.attachedFiles`；拖拽时同时 `attachFile` 与 `pendingContextFiles.add`。**发送只清空 `pendingContextFiles`，不会自动从 `attachedFiles` 移除**，因此用户可能仍看到 chip，但这**不代表**下一轮还会往 API 里再带一遍 `<context_files>`。

### 2.3 「当前笔记」`<current_note>` 也只在首轮附加

`shouldSendCurrentNote` 在首次标记已发送后为 false，后续轮次不再把 `currentNotePath` 放进 `turnRequest`（与 `markCurrentNoteSent` 的配合见 `FileContextState` / `InputController`）。这与拖拽队列「只送一次」的设计一致。

### 2.4 `/compact` 等命令

若为 `/compact`，构建请求时不会附带上述上下文（与 `isCompact` 分支一致）。Compaction 之后上下文由 SDK/摘要策略接管，与「附件是否重复注入」是另一条链路。

## 3. 「重复读入」应拆成三层理解

### 3.1 插件是否每一轮都把「附件路径块」再发一遍？

- **拖拽附加**：**不会**（第 2 轮起 `contextFiles` 为空，除非用户再次拖拽或存在其他入口重新入队）。
- **若用户每一轮都在输入框里手动 `@` 同一文件**：新消息的**可见文本**里可能重复出现提及；是否转成路径、是否额外注入，取决于提及解析与输入内容，**不属于**「chip 常驻自动重复注入」的同一路径。

### 3.2 模型上下文里是否还能「看到」这份文档？

- **对话历史**：第 1 条用户消息若已包含 `<context_files>`（及当时用户问题），在**未 compact、未截断**的情况下，后续轮次请求里仍会带上**整条历史**，因此**路径与首轮表述**仍可能在上下文中出现——这是**历史保留**，不是插件在第二轮重新追加一块新的 `<context_files>`。
- **文件正文**：`<context_files>` 在 Claudian 里主要是**路径列表**（见 `formatContextFilesLine`），**不等于**每次自动把整篇文档正文嵌入 prompt。正文通常通过模型侧 **Read 等工具**读入，读到的内容出现在**该轮及之后的助手/工具结果**中；是否**再次调用 Read** 由模型决定，**不是** Claudian 每轮强制把文件再读一遍。

### 3.3 Token / 成本角度

- 第二轮起**不会**因「同一拖拽附件」再额外增加一整段新的 `<context_files>` 块（队列已空）。
- 但若上下文携带**完整历史**，首轮用户消息（含 `<context_files>`）仍会占用窗口长度；若历史中有大段工具返回的正文，也会持续占用，直到 compact 或窗口滑动。

## 4. 直接回答用户原问题

1. **会不会像第一轮那样，每一轮都重新把「附件队列」里的文档再打包发一遍？**  
   **不会**（拖拽场景下，第二轮起队列为空，设计即为避免重复注入）。

2. **模型算不算「又读了一遍文档」？**  
   - **插件不会**每轮自动重新附加路径块或自动重新 Read 文件。  
   - **模型可能**在后续轮次**再次**调用 Read（例如用户新问题需要核对原文），这是模型行为，不是「附件常驻」导致的固定重复。

3. **界面上附件还在，是否表示下一轮还会发送？**  
   **不一定**。chip 可保留用于展示或用户心理预期，但**投递层**以 `pendingContextFiles` / `contextFiles` 为准。

---

## 5. 建议（面向使用者）

- 若希望后续轮次**明确**再带路径，可**再拖一次文件**或在新消息中 **@** 该文件（注意这可能增加重复路径文本）。
- 若对话很长、担心窗口占满，可关注 **/compact** 或会话续写策略对历史的影响（compact 后细节可能以摘要形式存在）。
- 与拖拽说明文档一致：删除 chip 可避免用户误以为仍会附带；实际是否附带以发送当刻队列为准（见《使用说明_拖拽文件添加上下文.md》中的表格说明）。
