# 上游社区信号调查报告

## 目标

面向「在官方仓库基础上维护自己 fork」的场景：用**只读采集 + 研判写作**得到一份周期性报告，回答：

- 上游 `main` 上还有哪些**你本地还没合入**的提交（代码缺口）？
- 社区 **Issues** 里有哪些值得关注的**风险/复现**信号？
- **Open PRs** 里有哪些**机会**与**维护成本**提示？
- 对你当前 fork：**哪些条目价值最高**（P0 / P1 / P2）及理由？

## 硬约束（必须遵守）

1. **禁止**修改 `src/`、`tests/`、`package.json`、`manifest.json`、`esbuild` 配置等业务与构建代码；**禁止**执行合并、cherry-pick、rebase、推送等会改动 Git 历史的操作（除非用户在本轮对话中明确要求改代码，本 Skill 默认不包含该授权）。
2. **允许**的操作：运行采集命令、`git fetch`/`git log`/`gh` 查询、在 `docs/04-Archives/` **新建或更新**调查报告类 Markdown。
3. 报告中的「待合并」必须以 **`HEAD..upstream/main`（或当前仓库实际的上游默认分支）** 为准，而不是凭感觉列上游「新闻」。

## 推荐执行顺序

### 步骤 1：原始采集（命令）

在仓库根目录执行（与 `package.json` 一致，使用 **npm**）：

```bash
npm run report:upstream-signals
```

可选：仅使用本地已有 remote 数据、不访问网络：

```bash
npm run report:upstream-signals -- --no-fetch
```

脚本会在 `docs/04-Archives/` 生成 `上游社区信号_采集_YYYY-MM-DD_HHMMSS.md`，内含 merge-base、提交列表快照、（若 `gh` 可用）Issues/PRs 列表。

记下该文件路径（脚本在标准输出打印完整路径）。

### 步骤 2：研判与定稿（智能体）

1. 阅读刚生成的**采集文件**全文。
2. 对照 **`docs/02-Areas/最佳实践_fork-同步上游与社区信号筛选.md`**，尤其第 2 节（代码缺口）、第 4 节（Issue/PR 怎么用）、第 7 节（检查清单）。
3. 在 **`docs/04-Archives/`** 新建定稿文件，建议命名：`上游社区信号调查报告_YYYY-MM-DD.md`（可与采集同日；若当日多份，可加后缀 `_2`）。

定稿正文建议结构：

| 章节 | 内容 |
|------|------|
| 摘要 | 本轮最重要 3～5 条结论（面向「我作为 fork 维护者」） |
| P0 高价值 | 强烈建议关注的提交/Issue/PR：与**当前缺口**、**你方在复现的问题**、**小而清晰的修复**相关 |
| P1 中价值 | 值得关注但可排期：与路线一致、需评估冲突或测试成本 |
| P2 低价值/观望 | 大功能、长期 Open、与本地路线不一致、文档-only 等 |
| 代码缺口清单 | 从采集中摘录关键提交，注明是否「已在共同祖先/等价实现」需用户核对 |
| 社区快照 | Issues/PRs 中与你环境（如 Windows、Obsidian 版本）相关的条目编号与一句话理由 |
| 建议的下一步 | 仅建议：例如「先 merge upstream」「只 cherry-pick 某 SHA」「继续观望某 PR」——**不代为执行** |

**价值排序原则（摘要）**

- 与 **`git log HEAD..upstream/main`** 直接相关的修复 / 已合并进上游的 PR 线 → 通常 **P0/P1**。
- 大量互动、明确复现、可能影响你方用户的 Issue → **P0/P1**（仍需对照你是否已含对应提交）。
- 与当前缺口无关的大功能 PR、或长期高维护成本的新提供商/协议 → 默认 **P2**。
- 纯文档、调查笔记 → 默认 **P2** 或「按需」。

### 步骤 3：可选 — 在原采集文件补充

若用户希望**单文件**：可在采集文件末尾「价值优先级研判（占位）」处填写完整研判，并在文首 YAML 或标题注明「已研判」。优先保持 `docs/04-Archives` 内可追溯（采集 vs 定稿分离更清晰）。

## 无 `gh` 时的处理

若环境无 GitHub CLI：仍可依采集文件中的 **Git 缺口**部分完成大半报告；Issues/PRs 章节注明「未拉取列表，请浏览器查看上游仓库」，并降低对社区条目的排序置信度。

## 与最佳实践的对齐（自检）

- [ ] 是否明确写了当前 `merge-base` 与 `HEAD..upstream` 分支名？
- [ ] 是否避免把「上游关闭的 Issue」自动等同于「我这边已好」？
- [ ] 是否区分「跟随上游 main 已合并」与「仍在 Open 的 PR」？
- [ ] 价值段落是否**按优先级**排列且每条有**一句人话理由**？
